# 网络分层与交易快速通道多轮测试报告

生成时间：2026-06-16

## 测试目标

本轮验证不做压测，只验证正确性和稳定性基础：

- 30 个验证者在相同 `epoch snapshot + slot + leader + fanout` 输入下，独立计算出完全一致的 Turbine 传播树。
- 每个节点都能知道自己的 `layer`、`parent_peer_id`、`child_peer_ids`，并且每个非 leader 节点只被一个父节点覆盖。
- 交易快速通道优先选择当前 slot 和未来若干 slot 的 leader，再补充 validator peer，状态接口可直接观测路由结果。
- 新增字段不破坏现有 RPC、mempool、配置、构建和全仓库测试。

## 本轮代码验证点

新增或强化的测试：

- `consensus.TestTurbineTreeEveryNodeComputesSameRoute`
  - 30 个验证者独立构造 Turbine 树。
  - 比对每个节点的 parent、layer、children。
  - 验证所有 follower 恰好有一个父节点。
- `cmd/posnode.TestTurbineAllNodesComputeSamePropagationTree`
  - 以 posnode 本地 consensus key 视角逐个计算位置。
  - 比对每个节点状态中的 parent/children 是否等于共识层参考树。
- `cmd/posnode.TestTransactionFastPathPrefersLeadersThenValidators`
  - 验证交易路由列表以当前和未来 leader 为前缀。
  - 验证开启 validator 旁路后，validator peer 会补齐到快速路径。
- `cmd/posnode.TestTransactionFastPathCanExcludeLocalPeer`
  - 验证状态展示和远端转发时不会把本节点 peer 放入待转发列表。

状态接口新增：

- `getNodeStatus.transaction_fast_path`
  - `start_slot`
  - `forward_slots`
  - `leader_slots`
  - `validator_peer_ids`

  - `fast_path_available`

metrics 新增：

## 6 台机器 × 每台 5 节点的正确性口径

用户提供的目标机器可以形成 30 节点网络：

- 本机：5 个节点。
- `192.168.181.136`：5 个节点。
- `192.168.181.137`：5 个节点。
- `192.168.181.138`：5 个节点。
- `192.168.181.139`：5 个节点。
- `192.168.120.223`：5 个节点。

实机正确性检查建议：

- 每个节点启用唯一 P2P 端口、唯一 RPC 端口、唯一 data_path。
- 30 个节点使用同一个 `chain_id`、同一个 `genesis_start_unix_millis`、同一个 genesis validator 列表。
- 每个节点的 `turbine_fanout` 保持一致，建议先用 `2` 或 `3`。
- 每轮采集 `getNodeStatus` 和 `getMetrics`，比对：
  - `head_height`
  - `head_slot`
  - `head_hash`
  - `finalized_height`
  - `current_leader`
  - `turbine.layer`
  - `turbine.parent_peer_id`
  - `turbine.child_peer_ids`
  - `transaction_fast_path.preferred_peer_ids`
  - `metrics.transaction_fast_path_peer_count`
- 稳定性不做压测，建议先跑 10 分钟，再跑 30 分钟，最后跑 2 小时。

## 当前结论

代码级 30 验证者确定性计算已经通过多轮测试；每个节点都能独立知道区块应该从谁来、应该传给谁。交易快速通道已经可观测，且测试证明路由顺序是 leader 优先、validator 兜底。

本轮没有实际启动 6 台机器上的 30 个进程。原因是仓库现有实机配置仍是 6 节点矩阵，且本地 400ms 配置未启用 RPC，不适合自动拉状态做 30 节点一致性比对。下一步应生成 30 份带 RPC 的配置，再进行实机正确性和稳定性验证。

## 追加测试：交易快速通道与区块分层传播稳定性

追加时间：2026-06-16

### 追加验证目标

- 交易能否稳定走快速通道：
  - 30 个节点在多个 slot 独立计算 `transaction_fast_path`。
  - 所有节点计算出的 leader-first preferred peer 列表必须一致。
  - 本节点 peer 必须从对外转发列表中剔除，避免交易回发给自己。
- 区块能否稳定走网络分层传播：
  - 30 个节点跨完整 epoch 内多个 slot 模拟 Turbine 逐层接力。
  - 从 leader 开始，只沿每个节点的 `child_peer_ids` 传播。
  - 每个 validator 必须最终收到一次，不能漏收、不能重复收到。

### 新增测试用例

- `cmd/posnode.TestTurbineLayeredPropagationReachesEveryValidatorAcrossSlots`
  - 30 节点。
  - `fanout = 3`。
  - 遍历 epoch 内全部 16 个 slot。
  - 每个 slot 从 leader 开始做 BFS 传播模拟。
  - 校验所有 validator 被覆盖，且每个 child 的 parent 与 Turbine 树一致。
- `cmd/posnode.TestTransactionFastPathStableAcrossSlotsForAllNodes`
  - 30 节点。
  - 遍历多个 slot。
  - 每个节点独立计算交易 preferred peer 列表。
  - 校验 preferred peer 列表以当前和未来 leader 为前缀，validator peer 全部存在，列表无重复。

## 追加实现：共识状态对外接口

追加时间：2026-06-16

### 目标

外部调用者需要直接知道：

- 节点在 Turbine 网络分层中的层级。
- 当前共识实际使用的 validator 权重。
- 已经生效的质押。
- 待生效的质押。
- 当前 epoch 中全部 validator 的同类信息。

### 接口变更

新增 JSON-RPC 方法：

```text
getConsensusStatus
```

`getNodeStatus` 新增字段：

```json
{
  "consensus": {
    "available": true,
    "slot": 2,
    "epoch_id": 0,
    "epoch_start_slot": 1,
    "epoch_end_slot": 16,
    "total_active_stake_lamports": 30000000,
    "local_validator_id": "...",
    "local_validator": {
      "validator_id": "...",
      "turbine_layer": 0,
      "effective_stake_lamports": 10000000,
      "weight_bps": 3333,
      "active_stake_lamports": 10000000,
      "pending_stake_lamports": 10000000,
      "activation_epoch": 1,
      "in_current_epoch": true
    },
    "validators": []
  }
}
```

字段含义：

- `effective_stake_lamports`：当前 epoch 共识真正使用的权重。
- `weight_bps`：该 validator 在当前 epoch 总权重中的占比，单位 bps。
- `active_stake_lamports`：链上 stake account 中已经 active 的质押桶。
- `pending_stake_lamports`：链上 stake account 中等待下个生效边界的质押桶。
- `in_current_epoch`：是否进入当前 epoch validator set。
- `in_turbine_tree`、`turbine_layer`、`turbine_child_peer_ids`：当前 slot 的网络分层位置。
