# 测试优化分析报告

生成时间：2026-06-15

## 1. 总体结论

本轮围绕 PoS 共识、真实账户交易、验证者加入、奖励结算、隐私交易和审计链路执行了三轮“测试 - 优化 - 复测”闭环。

结论：

- 本机自动化测试已达到当前代码级发布门禁要求。
- 账户创建、普通转账、交易提交、leader 选举、出块、验证者验证、QC 形成、奖励结算、slash/jail、四类隐私交易、审计准确性均通过测试。
- 第二轮测试驱动发现一个真实生产入口问题：`cmd/posnode.addTransaction` 未在交易进入 mempool 前验证签名。
- 已修复该问题：无效签名交易现在会被拒绝入池，并计入 `transactionsDrop` 指标。
- 当前仍未完成远程 5 机实机长稳验证。BLS 聚合签名在 2026-06-16 追加优化中已通过 CIRCL 实现，详见第 8 节。

第一轮优化：

- 增强 `cmd/posnode` mempool 边界测试。

复测命令：

```powershell
go test ./cmd/posnode -run "TestAddTransaction|TestSelectMempool" -count 10
```

结果：通过。

### 第二轮：随机顺序与安全入口检查

执行命令：

```powershell
go test ./... -shuffle=on -count 2
go test ./consensus ./runtime ./cmd/posnode ./blockchain -cover
```

结果：全部通过。

发现问题：

- `cmd/posnode.addTransaction` 原先只调用 `transaction.Validate()`。
- `Validate()` 只验证交易结构、签名数量和消息格式，不验证签名真实性。
- 结果是伪造签名交易可以进入 mempool，占用节点资源，并可能被继续广播。
- 虽然后续 runtime 执行阶段会拒绝无效签名交易，但生产入口应在 mempool 前置拒绝。

修复：

- 新增测试：`TestAddTransactionRejectsInvalidSignature`。
- 在 `addTransaction` 中调用 `transaction.HasValidSignatures()`。
- 无效签名交易直接拒绝。
- 验签失败和验签错误均增加 `transactionsDrop` 指标。

复测命令：

```powershell
go test ./cmd/posnode -run "TestAddTransaction|TestSelectMempool" -count 10
go test ./cmd/posnode -cover
```

结果：通过。

## 4. 修复点分析

### 问题：mempool 入口未验签

风险等级：高。

影响：

- 恶意节点可以发送大量结构合法但签名无效的交易。
- 节点会消耗 mempool 容量、排序资源、持久化资源和广播资源。
- 在高并发情况下会影响真实用户交易入池。

修复方式：

- 在 `cmd/posnode.addTransaction` 中加入签名真实性验证。
- 失败时拒绝入池。
- 更新指标，便于观察攻击或异常流量。

验证：

- `TestAddTransactionRejectsInvalidSignature` 已覆盖。
- `go test ./cmd/posnode -run "TestAddTransaction|TestSelectMempool" -count 10` 通过。
- 最终全量测试和构建通过。

### 5.2 BLS 聚合签名状态

本节为 2026-06-15 的历史风险记录；2026-06-16 已引入 CIRCL BLS12-381 并实现聚合 QC，详见第 8 节。

原始原因：

- Go 标准库没有 BLS12-381、pairing、hash-to-curve 和聚合签名验证。

影响：

- 2026-06-15 时 QC 不是高性能 BLS 聚合签名版本。
- 2026-06-16 代码路径已升级，但大规模验证者数量下仍需真实多机压测评估。

建议：

- 引入稳定 BLS 库。
- QC 格式升级为 `aggregate_signature + voter_bitmap + block_hash + slot`。
- 收到 QC 后按 bitmap 找公钥集合，一次聚合验签。

## 8. 2026-06-16 追加优化与门禁状态

本轮继续按生产建议做增量优化，重点覆盖 stake 生效模型、BLS 聚合 QC、metrics、treasury 安全边界。

已完成优化：

- stake 生效模型：`RegisterValidator` 和追加 `Stake` 写入 pending stake，并通过 `activation_epoch` 在下个 epoch 生效；`Unstake` 通过 `deactivation_epoch` 保证当前 epoch 权重在节点重启后仍一致，下个 epoch 才退出有效权重。
- effective stake：validator set 不再直接使用账户内 `ActiveStake + PendingStake`，而是统一通过 `EffectiveStakeAtEpoch` 计算，保证所有节点按同一 epoch 输入得到同一权重。
- BLS QC：新增 BLS key 生成、单票签名、单票验签、聚合签名、voter bitmap、聚合 QC 验证和二进制序列化兼容逻辑。
- BLS 上链绑定：stake account 新增尾部兼容字段 `BLSPublicKey`，新注册验证者和创世验证者都会写入 BLS 公钥；epoch snapshot hash 纳入 BLS 公钥。
- VoteCollector：新增 `AddVoteWithBLS`，达到 quorum 时直接产出带 `aggregate_signature` 和 `voter_bitmap` 的 QC。
- P2P vote envelope：Ed25519 签名验证身份，BLS 签名进入 QC 聚合；如果链上 validator 有 BLS 公钥，缺少或不匹配 BLS proof 会被拒绝。
- RPC metrics：新增 `getMetrics`，暴露当前 slot、leader、高度、状态根、区块哈希、QC 哈希、mempool、fork/reorg 和投票速率。
- treasury 安全：生产模式必须配置 `treasury_key_path`；非生产可显式禁用 hardcoded treasury；keystore 支持 seed、32 字节 seed base64、64 字节 secret key base64。
