# 三轮测试驱动优化报告

生成时间：2026-06-16

## 第一轮：BLS QC 权重裁决

### 原始测试信息

基线命令：

```powershell
go test ./... -shuffle=on -count=1
```

基线结果：通过。关键原始输出：

```text
ok  	solana_golang/p2p	8.260s
ok  	solana_golang/consensus	0.537s
ok  	solana_golang/cmd/posnode	0.841s
ok  	solana_golang/programs/stake	0.341s
```

红测命令：

```powershell
go test ./consensus -run TestVerifyBLSAggregateWithStakeRejectsInflatedStake -count 1
```

红测结果：失败。关键原始输出：

```text
FAIL	solana_golang/consensus [build failed]
consensus\bls_qc_test.go:211:8: undefined: VerifyBLSAggregateWithStake
```

### 问题分析

BLS 聚合签名只能证明 bitmap 中的验证者确实签过指定 `block_hash + slot`，但原实现没有把 QC 声明的 `confirmed_stake`、`threshold_stake` 与 epoch snapshot 的 stake 表重新计算绑定。

风险：

- 低 stake 验证者可以形成有效 BLS 签名。
- 恶意 QC 可以伪造更高的 `confirmed_stake`。
- 如果节点只验证聚合签名，不验证 stake 权重，就可能接受未达 2/3 权重的 QC。

### 优化内容

- 新增 `VerifyBLSAggregateWithStake`。
- 按 `validator_order + voter_bitmap` 还原 voter 集合。
- 按 epoch snapshot 的 `StakeMap` 重新计算总 stake、阈值 stake、confirmed stake。
- 要求：
  - `certificate.ThresholdStake == quorum.RequiredStake(totalStake)`
  - `certificate.ConfirmedStake == sum(voter stake)`
  - `confirmedStake >= requiredStake`
- `cmd/posnode.verifyQuorumCertificate` 改为调用带 stake 裁决的验证函数。

### 复测信息

复测命令：

```powershell
go test ./consensus -run "TestVerifyBLSAggregateWithStakeRejectsInflatedStake|TestVoteCollectorFormsBLS|TestBLS" -count 1
```

复测结果：通过。关键原始输出：

```text
ok  	solana_golang/consensus	0.673s
```

### 第一轮结论

第一轮解决的是共识安全问题，不是性能问题。修复后，外部广播来的 BLS QC 必须同时满足签名有效和 stake 权重一致，避免低权重签名伪造成高权重 QC。

## 第二轮：BLS QC 降级攻击防护

### 原始测试信息

红测命令：

```powershell
go test ./cmd/posnode -run TestVerifyQuorumCertificateRejectsUnsignedQCWhenBLSComplete -count 1
```

红测结果：失败。关键原始输出：

```text
--- FAIL: TestVerifyQuorumCertificateRejectsUnsignedQCWhenBLSComplete (0.00s)
    qc_verify_test.go:28: verifyQuorumCertificate() error = nil, want unsigned QC rejection
FAIL
FAIL	solana_golang/cmd/posnode	0.946s
```

### 问题分析

第一轮补了 BLS 聚合签名和 stake 权重校验，但节点接收外部 QC 时仍有一个降级风险：

- 如果 QC 没有 `signature_scheme`，旧逻辑直接返回成功。
- 当 epoch snapshot 中所有验证者都有 BLS 公钥时，网络已经具备聚合签名能力。
- 此时继续接受无 aggregate signature 的旧式 QC，会绕过 BLS 验签和 bitmap 权重裁决。

这不是兼容问题，而是生产网络中的降级攻击面。兼容旧 QC 只应发生在 validator set 未完整具备 BLS 公钥的情况下。

### 优化内容

- `cmd/posnode.verifyQuorumCertificate` 先读取 QC slot 对应的 epoch snapshot。
- 计算 `len(BLSPublicKeys) == len(Validators)`。
- 如果验证者集合 BLS 公钥完整，则无 `signature_scheme` 的 QC 直接拒绝。
- 如果验证者集合不完整，保留旧式 QC 兼容路径。

### 复测信息

复测命令：

```powershell
go test ./cmd/posnode -run "TestVerifyQuorumCertificateRejectsUnsignedQCWhenBLSComplete|TestHTTPJSONRPC|TestTreasury" -count 1
```

复测结果：通过。关键原始输出：

```text
ok  	solana_golang/cmd/posnode	0.974s
```

### 第二轮结论

第二轮修复的是协议降级问题。BLS 完整网络中，外部 QC 必须是聚合 QC；节点不再接受无签名旧式 QC，从而避免攻击者绕开 BLS 聚合验证。

## 第三轮：生产配置安全门禁

### 原始测试信息

红测命令：

```powershell
go test ./cmd/posnode -run "TestNormalizeNodeConfigRejectsProductionInsecureP2P|TestNormalizeNodeConfigAllowsProductionSecureP2P" -count 1
```

红测结果：失败。关键原始输出：

```text
--- FAIL: TestNormalizeNodeConfigRejectsProductionInsecureP2P (0.00s)
    config_test.go:12: normalizeNodeConfig() error = nil, want production insecure p2p rejection
FAIL
FAIL	solana_golang/cmd/posnode	1.452s
```

### 问题分析

P2P Host 层已经会拒绝生产模式下的 insecure P2P，但 `cmd/posnode` 配置归一化阶段没有提前拦截。风险在于：

- 配置文件看似合法，直到启动 P2P 时才失败。
- 运维排障会把错误定位到网络启动，而不是配置门禁。
- `Environment=production/prod` 与 `Production=true` 的安全要求不完全一致。

生产配置应尽早失败，错误必须明确。

### 优化内容

- 新增 `isProductionNodeConfig`，统一识别：
  - `production=true`
  - `environment=production`
  - `environment=prod`
- `normalizeNodeConfig` 在生产模式下强制：
  - 必须配置 `treasury_key_path`
  - `allow_insecure_p2p` 必须显式为 `false`
- `allowHardcodedTreasury` 也改为使用统一生产模式判断。

### 复测信息

复测命令：

```powershell
go test ./cmd/posnode -run "TestNormalizeNodeConfigRejectsProductionInsecureP2P|TestNormalizeNodeConfigAllowsProductionSecureP2P|TestTreasury|TestHTTPJSONRPC" -count 1
```

复测结果：通过。关键原始输出：

```text
ok  	solana_golang/cmd/posnode	0.964s
```

### 第三轮结论

第三轮把安全失败前移到配置加载阶段。生产模式不会再默认使用 insecure P2P，也不会在 `environment=prod/production` 时绕过 treasury 安全策略。

## 三轮总体结论

本轮三次测试驱动优化分别关闭了三个问题：

1. BLS QC 权重伪造：低 stake 签名不能伪造成高 stake QC。
2. BLS QC 降级攻击：BLS 完整网络不再接受无 aggregate signature 的旧式 QC。
3. 生产配置门禁：生产模式必须 secure P2P，且 `prod/production` 环境同样禁止 hardcoded treasury。

当前代码级门禁通过，但生产级门禁仍缺 Linux/Mac race 和真实 5 机长稳/故障注入数据。

## 追加优化：生产门禁与多节点一致性巡检

### 风险闭环目标

本次针对报告中仍未关闭的两个生产风险补充可执行门禁：

1. Windows 本机 `go test -race ./...` 受 ThreadSanitizer 内存映射限制影响，不能作为并发安全结论。
2. 真实多节点长稳验证缺少统一采样脚本，无法稳定比较 slot、height、leader、state_root、block_hash、qc_hash、validator 权重和 Turbine/快速通道路由。

### 优化内容

新增 `tools/run_posnode_gate.ps1`：

- 统一执行 `go test`、`go vet`、`go build ./cmd/posnode`。
- Windows 下默认跳过 race 并输出明确告警，避免把 TSan 平台失败误判为业务失败。
- 使用 `-RequireRace` 时，Windows 会以非零退出码阻断发布，强制把 race 门禁迁移到 Linux/macOS。
- 构建结束后自动清理 `posnode.exe` / `posnode`，避免临时产物污染工作区。

新增 `tools/check_posnode_consistency.ps1`：

- 支持通过 `-RpcUrls` 或 `-RpcUrlFile` 检查多节点。
- 每轮采集 `getNodeStatus`、`getMetrics`、`getConsensusStatus`。
- 比较 head/finalized 高度、区块哈希、状态根、QC 哈希、epoch、总生效质押、validator 权重指纹。
- 比较 Turbine 分层路由指纹，确认相同 slot 下节点计算出的区块传播路径一致。
- 比较交易快速通道 leader 指纹，确认交易提交路径能稳定指向当前/未来 leader。
- 支持输出 JSON 采样报告，方便后续归档到 ELK/Loki/Prometheus 或测试报告。

### 当前结论

代码级门禁已可一键复现，且本机测试、vet、posnode 构建通过。

生产级结论仍不能声明完全通过，原因如下：

- Linux/macOS 上的 `go test -race ./...` 尚未在目标机器完成并归档。
- 5 机/30 节点长稳测试尚未提供 RPC URL 文件和连续采样结果。
- 断网、延迟、乱序、节点重启、缺块同步、leader 不在线、fork/reorg 等故障注入仍需要在真实网络中跑满并归档。
