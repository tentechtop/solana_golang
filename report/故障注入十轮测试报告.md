## 测试驱动优化

预跑 `cmd/posnode` 缺块同步 handler 时暴露错误：

```text
p2p: invalid message: p2p: invalid peer id: utils: invalid base58 character '-' at position 4
```

原因：测试夹具里 `P2PPeerID` 使用 `"peer-consensus-status-a"` 这种字符串，业务状态展示能用，但一旦走 `p2p.NewResponseMessage` 就会触发真实 peer id 校验。

修复：

```go
P2PPeerID: mustRawKeyPair("consensus-status-peer-" + string(rune('a'+index))).peerID,
```

同时移除 `TestFaultInjectionMissingBlockHeightReturnsNotFound` 中的局部 peer id 覆盖，确保后续协议测试直接依赖真实夹具。

## 分析

当前本地故障门禁验证了核心状态机行为：乱序消息不会乱序投递，慢写会超时并上报错误，连接关闭不会静默吞掉写失败，缺父块 proposal 会进入 orphan 缓存，缺高度请求返回 `found=false`，leader 离线能形成 skip QC，reorg 后重启能恢复 fork tip，finalized 以下重组被拒绝。

Turbine 和交易快速通道测试验证了所有节点可计算同一传播树、leader 只分发给直接子节点、交易优先发给未来 leader 和验证者集合。质押、投票、奖励测试验证了验证者生命周期、奖励结算和处罚路径的主要业务规则。

## 剩余风险

本报告是本地确定性故障注入，不等价于 6 台机器、30 节点真实长稳网络结果。进入 stage 前建议补一轮真实环境测试：

1. 6 台机器每台 5 个节点，连续运行至少 2 小时。
2. 使用 Linux `tc netem` 注入 100ms、300ms、1s 延迟和 1% 丢包。
3. 使用防火墙临时切断 leader、验证者、普通 full node 的 P2P 端口。
4. 随机重启节点，确认缺块同步、orphan 重试、state root 恢复一致。
5. 每 10 秒采集所有节点 `slot`、`height`、`leader`、`state_root`、`block_hash`、`qc_hash`、`finalized_height`，全量比对。
6. 真实提交质押、撤质押、再次质押、普通转账和 RPC 查询，确认交易快速通道稳定命中未来 leader。
7. 在 Linux 或 macOS 上补跑 `go test -race .\...`。本机 Windows 结果为 ThreadSanitizer 分配失败，原始摘要见 `race_windows_failure.log`。

当前结论：本地门禁已满足进入受控 stage 环境的条件；生产发布仍需要真实多机长稳和网络故障注入报告。
