### 第二轮：随机顺序、覆盖率和安全入口检查

执行：

```powershell
go test ./... -shuffle=on -count 2
go test ./consensus ./runtime ./cmd/posnode ./blockchain -cover
```

结果：全部通过。

发现问题：

- `cmd/posnode.addTransaction` 只做交易结构校验，没有在进入 mempool 前验证签名。
- 风险：伪造签名交易可以占用 mempool 和广播资源，虽然后续执行阶段会失败，但生产入口不应放行。

优化：

- 增加 `TestAddTransactionRejectsInvalidSignature`，先确认当前实现会失败。
- 在 `addTransaction` 中调用 `transaction.HasValidSignatures()`。
- 无效签名交易拒绝入池，并增加 `transactionsDrop` 指标。

复测：

```powershell
go test ./cmd/posnode -run "TestAddTransaction|TestSelectMempool" -count 10
go test ./cmd/posnode -cover
```

结果：通过。

## 五、覆盖分析

### 1. 用户创建账户

覆盖测试：`TestTransactionSimulatorCreatesAccount`

验证点：

- payer 和 new account 均签名。
- system create account 指令扣除 payer 资金。
- new account 写入 owner、space 和租金豁免余额。
- 执行结果写集包含 payer 与 new account。

结论：账户创建链路可执行，余额与账户元数据写入正确。

### 2. 用户提交交易

覆盖测试：

- `TestAddTransactionAcceptsSignedUserTransaction`
- `TestHTTPJSONRPCSubmitsSignedTransaction`
- `TestHTTPJSONRPCRejectsInvalidSignedTransaction`

验证点：

- 用户签名的 transfer transaction 能进入 `cmd/posnode` mempool。
- HTTP JSON-RPC `sendTransaction` 能提交签名交易并返回交易签名。
- HTTP JSON-RPC `getBalance` 能查询创世资金账户余额。
- HTTP JSON-RPC `getHealth` 能返回节点健康状态和 mempool size。
- 重复提交同一交易不会重复入池。
- `transactions_in` 指标只增长一次。
- 无效签名交易通过 HTTP JSON-RPC 入口提交时被拒绝，且不会进入 mempool。
- 无效签名拒绝会计入 `transactions_drop`。

结论：节点本地交易提交入口、基础 HTTP JSON-RPC 提交路径、去重和 mempool 写入可用。

### 3. Leader 选举、出块、验证者验证

覆盖测试：

- `TestPoSRealAccountStakeVoteAndBlockFlow`
- `TestVoteCollectorBuildsQuorumCertificate`

验证点：

- epoch snapshot 根据 validator set 构建。
- leader schedule 能为 slot 选出 leader。
- leader 使用共识私钥生产区块。
- 验证者复算区块交易、状态根、签名。
- 投票按 stake 聚合形成 QC。
- 下一 epoch 使用链上 stake account 生成 validator set。

结论：单进程闭环内的 leader 选举、出块、验证、投票和 QC 形成通过。

### 4. 验证者加入必须走链上质押交易

覆盖测试：`TestValidatorJoinsOnlyAfterRegisterStakeTransaction`

验证点：

- 初始状态中 validator account 为空时，不会进入链上 validator set。
- 只有 `RegisterValidator` 交易进块并写入 stake account 后，新节点才进入下一个 validator set。
- 质押金额等于注册交易中的 stake lamports。

结论：验证者加入不是内存硬塞，必须通过链上注册/质押交易。

### 5. 奖励、投票积分与扣罚

覆盖测试：

- `TestApplyBlockRewardsCreditsLeaderFee`
- `TestApplyBlockRewardsSettlesVoteCreditsAtEpochBoundary`
- `TestApplyBlockRewardsJailsAndSlashesMissedVotes`

验证点：

- 区块交易手续费的 validator fee 进入当前 leader 账户。
- 有效 QC 形成 vote credits。
- epoch 边界按 commission bps 结算 validator 和 staker 奖励。
- missed vote 超阈值后触发 jail 和 slash。
- 奖励和扣罚在状态更新中一次性返回，供区块状态根复算。

结论：奖励、佣金、投票积分和扣罚的确定性执行通过。

### 6. 隐私四类交易与审计

覆盖测试：

- `TestAuditablePrivacyAccountSendsFourTransactionTypes`
- `TestAccountTransparentAndPrivateAuditBusinessLoop`
- `TestTransactionSimulatorDepositsTransparentToPrivate`
- `TestTransactionSimulatorWithdrawsPrivateToTransparent`
- `TestTransactionSimulatorTransfersPrivateToPrivate`
- `TestTransactionSimulatorAuthorizesPrivacyAudit`
- `TestFixedConfidentialInstructionExecutorCompletesFourTransactionTypes`

验证点：

- 透明到透明：普通 system transfer 成功，目标余额增加。
- 透明到隐私：deposit 成功，生成 privacy note 和监管审计记录。
- 隐私到隐私：private transfer 成功，源 note 标记 spent，生成 output note。
- 隐私到透明：withdraw 成功，透明账户收到隐私余额。
- 审计记录可被监管审计方解密。
- 审计 payload 中的 transaction type、amount、slot、commitment、nullifier、output commitment 与预期一致。
- 错误 proof 和错误 audit ciphertext 会被拒绝。

结论：四类隐私交易和监管审计准确性通过。

## 六、未覆盖项与风险

1. 远程多机实测未覆盖

本轮没有在以下机器上实际部署并启动节点：

- `192.168.181.136`
- `192.168.181.137`
- `192.168.181.138`
- `192.168.181.139`
- `192.168.120.223`

因此，本报告不能证明真实网络条件下的 NAT、防火墙、时钟漂移、磁盘延迟、长时间运行稳定性。

2. BLS 聚合签名未覆盖

当前 Go 标准库没有 BLS12-381、pairing、hash-to-curve 和聚合签名验证能力。本轮没有引入第三方 BLS 库，因此 QC 还不是 BLS 聚合签名版。

3. HTTP RPC 实例未实机压测

本轮已补充基础 HTTP JSON-RPC 端到端测试，覆盖 `getBalance`、`sendTransaction`、`getHealth`，并验证无效签名交易拒绝入池。
但还没有启动真实监听端口做长时间压测，也没有覆盖批量请求、并发请求、限流和真实 P2P 广播。

4. Race 检测环境受限

此前尝试 `go test -race ./blockchain ./runtime ./consensus` 时，Windows ThreadSanitizer 内存分配失败，错误码 `87`。普通测试通过，但 race 检测需在 Linux 环境重新跑。

## 七、生产建议

1. 远程 5 机部署后，至少运行 30 分钟以上，采集每个节点的 `slot`、`height`、`leader_id`、`block_hash`、`qc_hash`、`vote_count`、`mempool_size`。
2. 每台机器时钟必须同步，建议先启用 NTP；400ms slot 对时钟漂移比较敏感。
3. RPC 压测需要覆盖 `treasuryTransfer`、`transfer`、`registerValidator`、`stake`、`getValidatorSet`、`getHealth`。
4. 多机测试必须验证新节点从无币状态开始，先由 treasury 转账，再提交 `RegisterValidator`，最后到下个 epoch 进入 validator set。
5. 生产级 BLS QC 需要引入稳定 BLS 库，并在 QC 中落地 `aggregate_signature + voter_bitmap + block_hash + slot`，收到 QC 后按 bitmap 取公钥集合做一次聚合验签。
6. 建议在 Linux 环境补跑 `go test -race ./...`，并把 race 结果作为发布门禁。
