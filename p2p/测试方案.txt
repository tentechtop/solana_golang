P2P 模块生产级测试方案

一、目标范围

本方案用于验证 P2P 模块是否满足生产级上线要求，重点覆盖持久化一致性、SecureSession 安全边界、多节点集成、攻击场景、性能压测、长稳运行和生产配置审查。

已通过的基础门槛：
1. go test ./p2p/... 必须通过。
2. go test -race ./p2p/... 必须通过。
3. 生产环境必须禁止 AllowInsecure，必须启用有效 SecureSession。

二、测试环境

1. 单机开发环境
   - Windows + MSYS2 MinGW-w64 GCC，用于 race 检测。
   - 命令：go test -race ./p2p/...

2. Linux 或 CI 环境
   - 用于长时间压测、网络延迟注入和资源监控。
   - 建议固定 Go 版本、CPU 核数、内存大小和系统参数，保证结果可复现。

3. 多节点集成环境
   - 至少 4 个节点：1 个 bootstrap 节点、2 个普通节点、1 个异常节点。
   - 每个节点使用独立 PeerID、SecureIdentity、端口和数据目录。

三、PeerStore 持久化测试

功能目的 + 实现原因：验证节点表读写删除一致性，防止重启后发现层状态丢失或污染路由表。

1. SavePeer 正常保存
   - 输入有效 Peer。
   - 验证数据库存在对应 key。
   - 验证二进制反序列化后字段一致。

2. LoadPeers 正常加载
   - 预置多个 Peer。
   - 验证 limit 生效。
   - 验证返回 Peer 均通过校验。

3. DeletePeer 正常删除
   - 保存后删除。
   - 验证再次加载不包含该 Peer。

4. 上下文取消
   - ctx 已取消时调用 LoadPeers、SavePeer、DeletePeer。
   - 验证返回 context 错误，不应继续执行数据库写入。

5. 坏数据防护
   - 数据库写入非法 Peer 二进制。
   - LoadPeers 必须返回带上下文的解码错误。

6. 写一致性
   - 并发保存同一个 Peer 的不同状态。
   - 验证最终状态可解释，不出现半写入、乱码或反序列化失败。

验收标准：
1. p2p/peerstore 必须有独立单元测试文件。
2. 普通测试和 race 测试全部通过。
3. 错误信息必须包含 peer id 或数据库操作上下文。

四、多节点集成测试

功能目的 + 实现原因：验证真实节点间连接、发现、消息和重连流程，防止单元测试覆盖不到生命周期问题。

1. 多节点启动
   - 启动 bootstrap、nodeA、nodeB、nodeC。
   - 所有节点启用 SecureSession。
   - 验证监听成功、PeerID 唯一、路由表初始化成功。

2. 节点发现
   - nodeA 连接 bootstrap。
   - nodeB、nodeC 通过 KAD 查询发现 nodeA。
   - 验证 routing table 包含预期节点。

3. 拨号和消息收发
   - nodeA 拨号 nodeB。
   - 发送 ping 或自定义协议消息。
   - 验证消息只被目标节点处理，payload 正确。

4. 断线重连
   - 主动关闭 nodeB 连接。
   - 验证连接状态变为 disconnected。
   - nodeA 重新拨号成功，状态恢复 connected。

5. SecureSession 失败路径
   - 使用错误 network id。
   - 使用错误 PeerID。
   - 使用损坏签名。
   - 验证握手失败、连接关闭、错误计数和日志符合预期。

验收标准：
1. 多节点集成测试可重复运行。
2. 无 goroutine 泄漏。
3. 所有节点 Close 后端口释放，进程无残留连接。

五、攻击场景测试

功能目的 + 实现原因：验证异常输入和恶意行为不会突破认证、加密、限流和资源边界。

1. 伪造 peer id
   - 使用 PeerID A，但握手公钥来自 PeerID B。
   - 预期：SecureSession 拒绝，返回身份不匹配错误。

2. network id 不一致
   - 两个节点使用不同 NetworkID。
   - 预期：握手失败，连接关闭。

3. 重放包
   - 捕获已成功解密的 SecurePayload。
   - 再次发送同一个 sequence。
   - 预期：Open 返回 unexpected sequence。

4. 乱序包
   - 先发送 sequence 1，再发送 sequence 0。
   - 预期：后续包被拒绝。

5. 大包攻击
   - 构造超过 DefaultMaxMessageSize 的消息。
   - 预期：读写路径拒绝，不分配超大内存。

6. 慢连接
   - 建立连接后不发送首帧或极慢发送。
   - 预期：握手超时，连接关闭，goroutine 不泄漏。

7. 握手洪泛
   - 大量并发连接只发半截握手。
   - 预期：MaxPendingInbound、MaxConnectionsPerIP、生效，节点保持可服务。

8. 协议队列打满
   - 高频发送低优先级协议消息。
   - 预期：队列满后丢弃或拒绝，读循环不被拖垮。

验收标准：
1. 所有攻击用例不能导致 panic。
2. 内存、goroutine、连接数在攻击结束后回落。
3. 关键拒绝路径必须有明确日志和指标。

六、性能压测

功能目的 + 实现原因：验证在高并发和长时间运行下，连接池、广播、KAD 和消息队列不会成为瓶颈。

1. 高并发连接
   - 并发 100、500、1000 个连接阶梯测试。
   - 指标：连接成功率、握手耗时 P50/P95/P99、失败原因分布。

2. 广播压测
   - 连接多个节点后广播消息。
   - 指标：广播耗时、成功节点数、失败节点数、写队列长度。

3. KAD 查询压测
   - 并发执行 FIND_NODE。
   - 指标：查询耗时、返回 peer 数、路由表更新耗时。

4. 长稳测试
   - 连续运行 24 小时。
   - 每分钟采集 goroutine 数、内存、连接数、消息吞吐、错误数。

5. 慢读慢写测试
   - 部分节点故意不读或低速读。
   - 验证写队列上限、写超时和连接关闭策略有效。

验收标准：
1. 无持续内存增长。
2. goroutine 数稳定，无明显泄漏。
3. P99 握手耗时和消息耗时低于业务设定阈值。
4. 队列满、连接满、限流均按预期返回明确错误。

七、生产配置审查

功能目的 + 实现原因：验证生产配置不会绕过安全边界，防止部署错误导致安全降级。

1. SecureIdentity
   - 私钥必须来自安全生成流程。
   - 公钥必须能反推校验 PeerID。
   - 私钥不得写入普通日志。

2. SecureSession
   - Production=true 或 P2P_ENV=production 时必须启用。
   - AllowInsecure=true 在生产必须启动失败。
   - NetworkID 必须按链或环境隔离。

3. QUIC TLS
   - QUIC 使用临时自签证书。
   - 客户端跳过证书链校验。
   - 安全原因：节点身份认证和业务加密统一由 SecureSession 保障，QUIC TLS 不作为信任边界。

4. 日志脱敏
   - 不允许输出私钥、共享密钥、AES key、完整恢复票据。
   - peer id、connection id、message id 可以记录。

5. 监控指标
   - 连接数、入站拒绝数、单 IP 拒绝数。
   - SecureSession 握手成功和失败数。
   - 消息读取、写入、拒绝、限流数。
   - 协议队列深度、写队列深度。

6. 密钥轮换
   - 明确节点长期密钥轮换流程。
   - 明确旧节点身份失效和黑名单策略。
   - 明确恢复票据过期和清理策略。

验收标准：
1. 生产配置模板必须显式设置 Production=true 或 P2P_ENV=production。
2. 任何安全降级配置必须在生产启动失败。
3. 日志和指标满足故障排查，不泄露敏感材料。

八、最终准入清单

1. go test ./p2p/... 通过。
2. go test -race ./p2p/... 通过。
3. PeerStore 单元测试通过。
4. 多节点集成测试通过。
5. 攻击场景测试通过。
6. 高并发压测通过。
7. 24 小时长稳测试通过。
8. 生产配置审查通过。
9. 关键错误日志和监控指标可观测。
10. 无 panic、无数据竞争、无 goroutine 泄漏、无持续内存增长。

生产测试
24h / 72h 长稳压测
多网络环境：高延迟、丢包、抖动、断网恢复
恶意节点测试：伪造 peer id、重放、握手洪泛、大包攻击、慢连接
监控指标接入：RSS、goroutine、队列深度、p99、失败率、peer blocked
密钥生成、存储、轮换、泄露恢复流程
生产配置固化：必须启用 SecureSession，禁止 AllowInsecure
