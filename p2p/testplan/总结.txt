P2P 非并发耗时分析与极限压测总结

测试日期：2026-06-15
测试目标：
1. 判断非并发、单次连接、单次请求的真实请求响应耗时。
2. 判断 5 台客户端同时压测时，本机服务端的极限 req/s。
3. 分析为什么内网高并发压测下响应耗时会达到数百毫秒。
4. 判断当前 P2P 设计是否能支撑 400ms 出块目标，以及后续架构方向。


一、最终结论

1. 非并发单次请求不是慢点。

   在干净重测中：

   | 客户端 | 模式 | 单次请求响应 |
   | --- | --- | --- |
   | Linux 192.168.181.136 | 冷连接 | 7.783 ms |
   | Linux 192.168.181.136 | 预热连接 | 1.223 ms |
   | Mac 192.168.120.223 | 冷连接 | 48.511 ms |
   | Mac 192.168.120.223 | 预热连接 | 17.881 ms |

   Linux 虚拟机到本机服务端，连接复用后的单次请求响应约 1.2ms。
   Mac 到本机服务端跨另一个网段，连接复用后的单次请求响应约 17.9ms。

   这说明之前极限压测看到的 258ms p50，不是单个包在内网中传输需要 258ms，而是高并发堆积后的排队耗时。

2. 5 台客户端极限压测结果稳定在约 1.94 万 req/s。

   正式 3 分钟压测结果：

   | 指标 | 数值 |
   | --- | ---: |
   | 协议 | TCP |
   | 模式 | stateless request/response |
   | 请求 payload | 256 B |
   | 客户端数量 | 5 |
   | 每客户端并发 | 1024 |
   | 总并发 | 5120 |
   | 持续时间 | 180s |
   | 成功请求 | 3,491,925 |
   | 失败请求 | 0 |
   | 总吞吐 | 19,379.48 req/s |
   | 加权 p50 | 258.064 ms |
   | 加权 p95 | 311.746 ms |
   | 加权 p99 | 396.003 ms |

   这个结果说明当前同步 request/response 路径在 5 台客户端压测下，服务端可以无失败地处理约 1.94 万次请求/秒。
   但这个测试模式会把大量请求同时压进服务端和客户端等待队列，所以响应时间被并发深度拉高。

3. 高并发下的 258ms p50 符合排队模型。

   用 Little's Law 估算：

   总并发 / 总吞吐 = 5120 / 19379.48 = 0.264s

   0.264s 约等于 264ms，和实测 p50 258ms 非常接近。

   所以高并发测试里的主要耗时不是网络 RTT，而是：

   - 客户端大量 worker 等待响应。
   - 服务端连接读循环、协议分发、handler、响应写回形成队列。
   - 客户端 response matching 等待对应 request id 返回。
   - 单机服务端 CPU、调度、锁竞争、内存分配和 socket buffer 共同限制吞吐。

4. 当前 request/response 模型适合控制面，不适合作为高吞吐数据面。

   如果每笔交易、每个块分片都走同步请求响应，并且发送方等待每次 response，那么在高并发下会自然形成排队。
   这会直接影响 400ms 出块窗口内的数据扩散。

   因此当前 P2P 模块的 request/response 应该主要用于：

   - 握手
   - 节点发现
   - 状态查询
   - 小规模控制命令
   - 拉取缺失数据

   高吞吐路径应该改为：

   - 异步 Send
   - 批量发送
   - 流式传输
   - 分片传播
   - 聚合 ACK
   - 背压控制
   - 管线化处理

三、非并发耗时分析

非并发测试只看一次请求，不做极限压测。
这里分为冷连接和预热连接：

- 冷连接：包含建立连接、P2P 安全会话握手、发送请求、服务端处理、返回响应。
- 预热连接：先建立并预热连接，再测一次请求响应，更接近长连接已存在时的真实业务请求耗时。

结果：

| 客户端 | 冷连接 | 预热连接 | 说明 |
| --- | ---: | ---: | --- |
| Linux 192.168.181.136 | 7.783 ms | 1.223 ms | 同 VMware 网段，链路短 |
| Mac 192.168.120.223 | 48.511 ms | 17.881 ms | 跨 WLAN/路由路径，链路更长 |

非并发路径中的耗时主要来自：

1. 冷连接阶段

   - TCP connect
   - secure session 握手
   - 协议初始化
   - 第一次 request/response

2. 预热连接阶段

   - request 编码
   - 加密/封包
   - socket write
   - 服务端 read loop
   - 协议分发
   - handler 执行
   - response 编码和写回
   - 客户端 read loop
   - request id 匹配和唤醒等待方

从结果看，Linux 预热连接 1.223ms，说明本机和同网段 VM 的常规请求响应路径并不慢。
Mac 预热连接 17.881ms 明显更高，主要是网络路径、无线/路由、不同网段和设备调度差异造成，不代表 P2P 模块单机处理必然需要十几毫秒。

| linux-138 | 706,753 | 3,921.85 | 257.87 ms | 309.76 ms | 371.34 ms | - |
| linux-139 | 707,340 | 3,927.02 | 258.02 ms | 304.25 ms | 364.92 ms | - |
| mac-223 | 659,027 | 3,656.93 | 259.36 ms | 341.61 ms | 469.25 ms | 2425.67 ms |

汇总：

| 指标 | 数值 |
| --- | ---: |
| 总成功请求 | 3,491,925 |
| 总失败请求 | 0 |
| 总吞吐 | 19,379.48 req/s |
| 加权 p50 | 258.064 ms |
| 加权 p95 | 311.746 ms |

校准测试显示：

| 每客户端并发 | 总并发 | 总吞吐 |
| ---: | ---: | ---: |
| 1024 | 5120 | 20,630.81 req/s |
| 2048 | 10240 | 18,759.29 req/s |
| 4096 | 20480 | 18,792.90 req/s |

每客户端并发从 1024 提升到 2048/4096 后，吞吐没有继续提升，反而略降。
这说明瓶颈已经不在“客户端并发不够”，而在服务端处理链路、Go runtime 调度、连接读写队列、锁竞争、内存分配、socket buffer 或本机 CPU 上。


五、为什么内网高并发响应仍然慢

高并发响应慢的核心原因是排队，不是单包网络慢。

在 5 客户端正式测试中：

总并发 = 5120
吞吐 = 19379.48 req/s
平均在系统中停留时间 = 5120 / 19379.48 = 0.264s

也就是只要保持 5120 个请求同时在途，而系统每秒只能完成约 19379 个请求，那么单个请求平均就要在系统里等约 264ms。
实测 p50 为 258ms，正好吻合。

因此这个 258ms 不是“网络 RTT”，而是“请求进入系统后等待被处理和返回”的总驻留时间。

换句话说：

- 非并发测试看的是一辆车过路口需要多久。
- 极限压测看的是 5120 辆车同时挤到路口后，平均每辆车多久通过。

当前同步 request/response 模型在极限并发下自然会出现这个现象。


六、一个请求最可能在哪些环节耗时最长

在非并发预热连接下，耗时最长的环节大概率不是网络传输，而是以下组合：

1. 客户端 request 写入到连接。
2. 服务端 read loop 读取并解包。
3. 服务端协议分发到 handler。
4. handler 生成 response。
5. 服务端写回 response。
6. 客户端 read loop 读取 response。
7. 客户端根据 request id 匹配 pending request，并唤醒等待方。

在极限压测下，最长耗时变成排队：

1. 客户端 worker 等待可用连接和响应。
2. 服务端连接读循环处理大量请求。
3. 服务端分发队列和 handler 执行队列堆积。
4. 服务端写响应时 socket buffer 和调度等待。
5. 客户端 response matching 中 pending request 堆积。

要精确判断最长环节，需要在代码里增加阶段耗时埋点。
建议记录以下时间点：

| 阶段 | 时间点 |
| --- | --- |
| 连接 | dial_start、connected、secure_done |
| 客户端发送 | request_encode_start、write_start、write_done |
| 服务端接收 | server_read_done、dispatch_start |
| 业务处理 | handler_start、handler_done |
| 服务端返回 | response_write_start、response_write_done |
| 客户端接收 | client_read_done、matched_pending_request |

有了这些字段，就可以把一次请求拆成：

总耗时 = 连接耗时 + 客户端写耗时 + 网络去程 + 服务端排队 + handler 耗时 + 服务端写耗时 + 网络回程 + 客户端匹配唤醒耗时


七、对 400ms 出块的影响判断

当前测试结果不能简单理解为“P2P 不满足 400ms 出块”。

原因：

1. 非并发单次请求在 Linux 同网段预热连接下只有 1.223ms。
2. 5 客户端 p50 258ms 是 5120 并发 request/response 下的排队结果。
3. 区块生产和传播不应该依赖每笔数据同步 request/response 等待返回。

真正的问题是：

如果把高吞吐交易传播或区块传播设计成同步 request/response，那么一定会遇到排队瓶颈。
这条路径不适合作为 400ms 出块的数据面。

400ms 出块需要的是：

- 提前转发交易。
- 异步传播。
- 批量传输。
- 分片传播。
- 不等待每条消息逐个响应。
- 接收端并行验证和并行执行。
- 网络、验证、执行、打包形成 pipeline。


八、Solana 方向对当前 P2P 的启发

Solana 的核心方向不是“每笔交易请求一次，等待一次响应”。
它更接近：

1. 交易提前转发给未来 leader。
2. 数据传播走批量和分片。
3. 网络接收、签名验证、银行执行、打包、广播形成流水线。
4. 执行层按账户锁并行处理。
5. 数据面避免同步请求响应成为主路径。

所以 Solana 也会遇到瓶颈，但瓶颈通常不在“单个 request/response 等待”上，而在：

- leader ingress
- 签名验证
- 账户锁冲突
- banking stage
- block packing
- shred 传播
- 磁盘和网络 IO
- validator 客户端实现效率

对当前项目来说，P2P 的设计方向应该是：

- control plane 保留 request/response。
- data plane 使用异步消息、批量、流式、分片和背压。
- 不要把高吞吐路径建立在每条消息等待 response 上。


九、建议的下一步优化

1. 增加单请求阶段耗时埋点。

   先不要猜锁竞争或网络慢，应该把一次请求拆开量化。
   最小改动是在 p2pstress 和 p2p request 路径里记录：

   - write_start/write_done
   - server_read_done
   - handler_start/handler_done
   - response_write_done
   - client_read_done
   - request_matched_done

2. 增加 1000 次非并发顺序请求测试。

   单次请求容易受抖动影响。
   建议补一个 sequential 模式：

   - 并发固定为 1。
   - 复用同一条连接。
   - 连续发送 1000 次 request/response。
   - 输出 min、avg、p50、p95、p99、max。

   这个结果比只测一次更能说明真实长连接请求耗时。

3. 新增 data-plane 压测模式。

   request/response 只能代表控制面性能。
   需要单独测试：

   - async send
   - batch send
   - stream
   - no per-message response
   - aggregate ack

   这样才能判断交易传播或区块分片传播能达到多少 msg/s、MB/s 和 tail latency。

4. 在 5 客户端压测期间采集 pprof。

   需要采集：

   - CPU profile
   - heap profile
   - block profile
   - mutex profile
   - goroutine dump

   目标是确认极限瓶颈到底在：

   - 加解密
   - 编码/解码
   - pending request map 锁
   - handler 调度
   - 连接写锁
   - Go runtime 调度
   - GC
   - socket IO

十一、最终判断

当前 P2P 模块的单次请求响应能力没有表现出无法接受的问题。
真正的问题是高并发同步 request/response 模型在达到吞吐上限后会产生明显排队。

因此结论是：

1. 非并发单请求耗时可以接受。
2. 5 客户端极限 request/response 吞吐约 1.94 万 req/s。
3. 高并发下 258ms p50 主要来自排队，不是内网单包传输慢。
4. request/response 不应该作为交易传播和区块传播主路径。
5. 400ms 出块需要异步、批量、分片、流水线和背压设计。
6. 下一轮优化应该先做阶段耗时埋点，再针对 pprof 和阶段数据优化真正瓶颈。
