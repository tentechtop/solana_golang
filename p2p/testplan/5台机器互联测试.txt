P2P 5台客户端互联极限压测报告
测试日期：2026-06-15
测试目标：使用 5 台远程客户端同时压测本机 P2P 服务端，验证 request/response 模式下的稳定极限 req/s、失败率和响应延迟。


1. 测试结论

本轮正式测试使用 TCP + stateless 无序调度，5 台客户端同时请求本机服务端，持续 180 秒。

最终结果：
- 总成功请求：3,491,925
- 总失败请求：0
- 总吞吐：19,379.48 req/s
- 总并发：5,120
- 加权 p50：258.1 ms
- 加权 p95：311.7 ms
- 加权 p99：396.0 ms

结论：
1. 当前 5 客户端压本机服务端的稳定零失败吞吐约为 1.94 万 req/s。
2. 单客户端约 3.65k-3.95k req/s，4 台 Linux VM 表现接近，Mac 客户端略低。
3. 校准测试显示每客户端 1024 并发是当前更优档位；2048 和 4096 并发没有提升总吞吐，反而提高延迟。
4. 该结果是 request/response 口径，一次请求包含客户端发送、服务端处理、服务端响应、客户端校验，实际双向消息吞吐约为 req/s 的 2 倍。
5. 本轮没有采集服务端 CPU、GC、pprof，因此只能判断端到端极限，不能精确归因到 CPU、网络、GC 或锁竞争中的某一项。

5. 并发校准结果

正式测试前，对每客户端 1024、2048、4096 三个并发档位进行 10 秒校准。

| 每客户端并发 | 总并发 | 总成功请求 | 失败 | 总 req/s | 结论 |
|---:|---:|---:|---:|---:|---|
| 1024 | 5120 | 210,748 | 0 | 20,630.81 | 短跑最高，延迟较低 |
| 2048 | 10240 | 195,929 | 0 | 18,759.29 | 吞吐下降，延迟升高 |
| 4096 | 20480 | 203,189 | 0 | 18,792.90 | 吞吐未恢复，延迟明显升高 |

校准结论：
- 当前拓扑下不是并发越高越快。
- 1024 并发/客户端可以让服务端保持较高吞吐，同时避免请求排队过深。
- 因此正式 3 分钟测试选择 1024 并发/客户端。

7. 性能分析

7.1 吞吐上限判断

本轮正式稳定吞吐约 19.38k req/s。由于 request/response 是双向通信，粗略换算消息处理量：
- 请求消息：约 19.38k msg/s
- 响应消息：约 19.38k msg/s
- 合计 P2P 消息读写：约 38.76k msg/s

该值还包含：
- secure session 加密和解密
- Message 序列化和反序列化
- TCP frame 读写
- request manager 等待响应和匹配 request_id
- 协议调度 worker 执行 handler
- 客户端 payload 校验和延迟统计

因此该结果不能和单向 UDP 包转发 TPS 直接对比。

7.2 延迟表现

整体加权 p50 为 258.1ms，p95 为 311.7ms，p99 为 396.0ms。

这个延迟不是网络 RTT 本身，而是在高并发下的完整端到端排队时间：
- 客户端 worker 排队
- 写队列排队
- 服务端连接读循环排队
- secure session 解密
- 协议 worker 调度
- handler 生成响应
- 响应写回
- 客户端读响应和匹配 request

当并发从 1024 提升到 2048 或 4096 时，总吞吐没有提升，说明瓶颈已经不是客户端 worker 数量，而更可能是服务端处理、连接读写、调度队列、GC 或网络路径。

7.3 Mac 客户端差异

Mac 客户端吞吐低于 Linux VM，p95/p99 更高，最大延迟明显更大。

可能原因：
1. Mac 走 192.168.120/121 局域网路径，Linux VM 走 VMware 192.168.181 网段，网络路径不同。
2. Mac 的调度、TCP 栈、无线/有线路径和本机到 Mac 的链路可能有额外抖动。
3. Mac 客户端日志中出现较大的 max_latency，但 failed=0，说明只是尾延迟尖峰，不是稳定错误。

7.4 当前瓶颈推断

仅根据端到端结果，较可能的瓶颈方向：
1. 服务端单进程 CPU/Go 调度达到平台上限。
2. request/response 模式中 request manager 和响应匹配带来额外同步成本。
3. secure session 加解密在高并发下占用 CPU。
4. 大量 goroutine + 高 write queue 带来 GC 和内存压力。
5. 本机同时承载 Windows 服务端、VMware 虚拟网卡和跨网段通信，网络栈路径较复杂。

不能确认的项：
- 服务端 CPU 是否打满。
- GC pause 和 heap 增长是否是主要瓶颈。
- 是否存在局部锁竞争。
- TCP socket buffer 是否达到系统限制。

需要 pprof、系统 CPU、网卡吞吐、GC trace 才能进一步确认。


8. 和单客户端测试对比

前一次单客户端 TCP stateless 峰值约 12.07k req/s。
本次 5 客户端正式 3 分钟稳定值约 19.38k req/s。

对比结论：
1. 多客户端确实提高了服务端总吞吐，说明单客户端存在压测侧瓶颈。
2. 5 客户端并没有线性提升到 5 倍，说明服务端或本机网络路径已经成为主要限制。
3. 3 分钟长跑更接近稳定工作区间，参考价值高于 10 秒短跑峰值。


9. 风险和局限

1. 本轮只测 TCP stateless，没有同时跑 QUIC 和 ordered。
2. 本轮服务端没有开启 pprof profile-dir，缺少 CPU/heap/mutex/block profile。
3. 本机服务端运行在 Windows，真实生产 Linux 服务端可能结果不同。
4. Linux VM 和 Mac 不在同一网络路径，客户端结果有环境差异。
5. p2pstress 日志和统计本身会消耗 CPU 和内存，极限值包含压测工具开销。
6. 测试 payload 为 256B，小包结果不能直接代表大交易、大区块或状态同步场景。


10. 后续建议

建议下一轮测试分三步：

1. 服务端剖析测试
   - 服务端开启 profile-dir。
   - 采集 CPU、heap、goroutine、mutex、block profile。
   - 同步采集 Windows CPU、内存、网卡吞吐。

2. 协议矩阵测试
   - TCP stateless
   - TCP ordered
   - QUIC stateless
   - QUIC ordered
   每组使用 5 客户端、180s、并发 512/1024/2048 三档。

3. 真实业务模型测试
   - 单向广播/通知吞吐。
   - 批量交易吞吐。
   - 不同 payload：256B、1KB、4KB、16KB。
   - 多服务端互联，而不是所有客户端只打一个服务端。
