# P2P 锁竞争导致的问题分析与优化方案

分析日期：2026-06-15

## 1. 核心结论

当前 P2P 模块没有证据显示存在严重的数据竞态，但存在明显的性能竞争。

这里必须区分两个概念：

- 数据竞态：多个 goroutine 无保护读写同一份数据，可能导致错包、串包、状态破坏。
- 性能竞争：多个 goroutine 都被锁、channel、队列、单 stream、单分区等待限制，数据是正确的，但吞吐上不去。

从现有测试结果看，数据正确性是稳定的：

- 无序协议正确性测试：566,281 条成功响应，0 失败，0 payload mismatch。
- 有序协议正确性测试：466,691 条成功响应，0 失败，0 payload mismatch。
- RequestID 没有错配表现。
- errors = {}。

因此当前问题不是“并发写坏数据”，而是“高压下同步点太多，单条消息成本高”。

## 2. 当前测试暴露的吞吐现象

现有极限 req/s 测试结果：

- 无序 stateless request/response：稳定约 18k-19k req/s。
- 有序 ordered request/response：稳定约 15k-15.6k req/s。
- 无序多 stream 比有序单 stream 高约 20% 左右。
- 无序并发继续升高到 4 VM * 256 后，没有获得明显吞吐增长，并出现 peer backoff 快速失败。

这些现象说明：

- QUIC 多 stream、并行解析、并行协议调度已经产生收益。
- 但是 request/response 完整链路仍然很重。
- 有序协议受单 peer + protocol 保序约束，天然比无序低。
- 当前 18k-19k req/s 不是 QUIC 理论上限，而是完整安全链路 + 虚拟机客户端 + 请求响应模型下的上限。

## 3. 完整 request/response 链路成本

当前压测即使标记为 stateless，也仍然是 request/response：

- 协议 HasResponse=true。
- 客户端调用 Host.Request。
- 每条请求注册 RequestID waiter。
- 服务端每条请求都回 response。
- 客户端每条响应都做 RequestID 匹配和 payload 字节级校验。

完整链路大致为：

1. 客户端创建 Message。
2. 克隆 payload，填充 FromPeerID、ToPeerID、RequestID。
3. RequestManager 注册 waiter。
4. 连接表查找当前 peer connection。
5. 消息编码。
6. SecureSession AES-GCM 加密。
7. QUIC/TLS 写入。
8. 服务端 QUIC 收包。
9. 服务端 SecureSession 解密。
10. 服务端协议解析和分类。
11. 协议调度进入 worker/partition。
12. handler 处理并构造 response。
13. response 再次编码、业务加密、QUIC/TLS 写回。
14. 客户端解密响应。
15. RequestManager fulfill。
16. 唤醒等待中的请求 goroutine。
17. 校验 payload。

这个链路里每一步都正确，但高 req/s 下会叠加出大量锁竞争、channel 竞争、调度开销、内存分配和 GC 压力。

## 4. 主要竞争点分析

### 4.1 RequestManager 等待表竞争

位置：p2p/request_manager.go

当前设计：

- requestManager 有 32 个 shard。
- 每条 request 调用 register，进入一个 shard mutex。
- response 回来后调用 fulfill，再进入同一个 shard mutex。
- 超时或返回后调用 unregister。
- 每条 request 都创建一个 waiter channel。

影响：

- request/response 每条消息至少触碰等待表 2 次。
- 高并发下同 shard 请求会排队。
- 32 分片在 18k-20k req/s 下已经能工作，但离 20 万 req/s 目标太少。
- waiter channel、defer unregister、context timeout 都增加调度和 GC 成本。

判断：

- 这是 request/response 模型的核心性能瓶颈之一。
- 它不是 bug，而是强确认语义的成本。

### 4.2 Host 全局连接表锁竞争

位置：p2p/host_connections.go、p2p/request_manager.go

当前设计：

- host.mutex 是全局 RWMutex。
- 连接查询、连接替换、连接移除、peer 状态读取都经过它。
- Host.Request 需要查找 connection。
- writePeerMessage 还会通过 currentConnectionForPeerWrite 再确认当前连接。

影响：

- 高频请求路径会频繁 RLock。
- 连接建立、连接替换、失败记录等写锁会阻塞所有读锁。
- 在连接稳定时，RLock 成本仍然会出现在每条请求路径上。

判断：

- 当前实现更偏正确性和简单性。
- 高吞吐下需要把“热路径 connection 查询”从全局 host.mutex 中拆出来。

### 4.3 SecureSession 加解密序列锁竞争

位置：p2p/secure_session.go

当前设计：

- sendMutex 保护发送序列号。
- receiveMutex 保护接收序列、replay/pending 检查。
- AES-GCM 加密解密本身也占 CPU。
- 当前已做过优化：接收侧尽量把 AES-GCM 解密移出大锁，只在序列检查和提交时持锁。

影响：

- 安全会话必须保证 nonce/sequence 不重复，不能随意去锁。
- QUIC 自身已有 TLS 1.3，加上业务层 AES-GCM，形成双层加密成本。
- request/response 下每条业务请求和响应都会走一次业务加解密。

判断：

- 这部分是安全正确性的必要成本。
- 继续优化必须谨慎，不能为了吞吐破坏 nonce 唯一性和 replay 防护。

### 4.4 QUIC stream 池和 stream 写锁竞争

位置：p2p/quic_transport.go

当前设计：

- QUICConnection 维护 high/normal/low read 队列。
- streamMutex 保护 stream 池注册、注销、选择。
- 每个 priorityStream 有自己的 mutex，保证同一 stream 写入不交错。
- stateless 可以使用同优先级多 stream 池。
- ordered 保持单 stream。

影响：

- 多 stream 已经降低了单 stream 阻塞。
- 但 stream 池选择和 stream 生命周期仍然需要锁。
- 每个 stream 的 WriteMessage 仍然要串行写。
- ordered 协议天然被单 stream 限制。

判断：

- 当前改造方向正确。
- 下一步要减少 stream 池热路径锁开销，或者让 stream 池更长期稳定，少注册/注销。

### 4.5 connection_writer channel 竞争

位置：p2p/connection_writer.go

当前设计：

- 写队列按 high/normal/low 拆分。
- 同一优先级又拆 serial/parallel lane。
- QUIC 支持多 worker 并行 flush。
- channel 作为有界背压队列。

影响：

- channel 本身是同步点。
- 高并发下大量 goroutine 同时向同一 priority queue 写入，会产生竞争。
- 队列保护了内存和慢连接，但也引入排队和调度成本。

判断：

- 有界队列是必要的，不能直接取消。
- 需要对不同语义消息拆路径：强确认走队列；高频 notify 可做批量 drain 或更轻量的发送路径。

### 4.6 协议调度分区和有序 sequencer

位置：p2p/protocol_dispatcher.go、p2p/host_connection_loop.go

当前设计：

- stateless 按 message/requestID 打散到不同分区。
- ordered 默认按 peer + protocol 保序。
- state-key 支持同 key 保序、不同 key 并行。
- connectionMessageSequencer 为 ordered 消息保序。

影响：

- ordered 协议必然牺牲吞吐换顺序。
- 如果把本可并行的业务错误标成 ordered，会直接造成吞吐下降。
- 如果把必须保序的业务错误标成 stateless，会导致状态一致性风险。

判断：

- 现在最大风险不是代码没有并行能力，而是协议语义必须准确分类。
- state-key 是后续提升有状态吞吐的关键。

### 4.7 消息编码和 buffer 分配导致 GC 压力

位置：message 编解码、SecureSession 加解密、压测 payload 构造路径。

当前问题：

- 每条消息会产生多个临时 []byte。
- 编码、加密、解密、payload clone 都可能分配。
- 高 req/s 下短生命周期对象会增多。
- Go GC 扫描和回收这些对象会消耗 CPU。

判断：

- 这不是锁竞争，但会放大调度和 CPU 压力。
- buffer pool 能降低 GC，但必须只复用临时 buffer，不能复用已经交给业务层或队列的 payload。

## 5. 为什么“只提高并发”不能根治

当前 4 VM * 192 已经能达到稳定高压点，继续提高到 4 VM * 256 后没有线性增长。

原因是：

- 并发只能填满流水线，不能降低单条消息成本。
- 当锁、channel、加密、GC、VM CPU 被打满后，提高并发只会增加排队。
- 排队会拉高 p95/p99 延迟。
- 延迟升高后更容易触发 timeout/backoff。

所以后续优化不能只调参数，必须减少热路径同步点和不必要的 request/response。

## 6. 根因判断

当前 req/s 低的根因不是单一锁，也不是单一加密成本，而是架构上把太多消息放进了完整 request/response 链路。

最核心的问题是：

- 高频无状态消息不应该每条都创建 waiter 并等待响应。
- 可批量查询不应该一个对象一个 request。
- 可按业务 key 并行的有状态消息不应该全部落到 ordered 单分区。
- 连接查询、请求等待表、加密序列、写队列、协议分区都在每条消息上叠加成本。

根治方向是把 P2P 拆成三条明确的数据路径：

1. 控制面：request/response，低频强确认。
2. 数据面：notify，高频无确认或异步确认。
3. 同步面：batch/stream，大对象和批量查询。

## 7. 协议语义分类建议

协议不应该只看“无状态/有状态”，而应该同时看：

- 是否需要返回数据。
- 是否需要强确认。
- 是否允许重复。
- 是否允许丢失后靠后续同步修正。
- 是否要求同 key 保序。
- 是否适合批量。

建议分类如下：

| 业务类型 | DeliveryMode | HasResponse | Concurrency | 说明 |
|---|---|---:|---|---|
| 心跳上报 | notify | false | stateless | 不需要每次 pong，活跃状态可由收包时间更新 |
| peer hints | notify | false | stateless | 节点提示可重复、可丢、可被后续覆盖 |
| node status | notify | false | stateless/state-key | 状态广播不应每条同步等待 |
| 交易广播 | notify | false | state-key | 按 tx hash / sender / shard 分区 |
| 区块广播通知 | notify | false | state-key | 通知可走单向，缺块再拉取 |
| find-node | request | true | stateless | 查询需要返回节点列表 |
| query block/header | batch request | true | stateless | 一次查多条，减少 waiter |
| get resource | batch/stream | true | state-key | 大对象建议分片或流式 |
| 指令下发 | request | true | ordered | 需要明确成功/失败 |
| 状态变更 | request | true | ordered/state-key | 同状态 key 保序 |
| 共识投票/QC | notify 或 request | 视共识语义 | ordered/state-key | 如果共识层已有确认，可不每条 request |

## 8. 需要增加的协议元数据

当前 ProtocolSpec 已有：

- HasResponse
- Priority
- Class
- Concurrency
- PartitionKey

建议补充：

```go
type ProtocolDeliveryMode uint8

const (
    ProtocolDeliveryNotify ProtocolDeliveryMode = iota
    ProtocolDeliveryRequest
    ProtocolDeliveryBatch
    ProtocolDeliveryStream
)

type ProtocolReliability uint8

const (
    ProtocolReliabilityBestEffort ProtocolReliability = iota
    ProtocolReliabilityRetry
    ProtocolReliabilityMustAck
)
```

目标：

- 用 DeliveryMode 决定调用 Send、Request、BatchRequest、Stream。
- 用 Reliability 决定是否需要重试、ack、去重、落盘。
- 用 Concurrency 决定 stateless、ordered、state-key。
- 用 HasResponse 保持兼容，但不再让它承担全部语义。

## 9. 优化验证方式

所有优化必须通过数据证明，不建议只凭感觉改。

建议新增三类压测：

1. request 模式：
- 现有模式。
- 用于测试强确认业务真实 req/s。

2. notify 模式：
- HasResponse=false。
- 客户端 Host.Send。
- 服务端 RegisterVoidHandler。
- 不创建 RequestID waiter。
- 用于测试无状态单向 msg/s 上限。

3. batch 模式：
- 一次 request 携带 N 个 item。
- 一次 response 返回 N 个 result。
- 用于测试同步类协议的真实吞吐。

同时开启画像：

- mutex profile：确认锁竞争热点。
- block profile：确认 channel 和队列阻塞热点。
- CPU profile：确认加密、编码、调度占比。
- heap profile：确认 message/payload/buffer 分配热点。

## 10. 风险边界

以下优化不能做错：

- 不能把必须强确认的指令改成 notify。
- 不能把必须保序的状态变更改成 stateless。
- 不能复用已经交给业务、队列、map、QUIC 写入路径的 payload buffer。
- 不能为了减少锁破坏 AES-GCM nonce 唯一性。
- 不能为了吞吐取消 backpressure，否则慢连接会拖垮内存。

正确性优先级高于吞吐。

## 11. 最终优化方案

### 第一阶段：先做语义拆分，立即降低不必要竞争

目标：把高频无状态流量从 request/response 重链路中移走。

任务：

1. 给 p2pstress 增加 `-stress-mode request|notify|batch`。
2. notify 模式注册 `HasResponse=false` 的 stress 协议。
3. notify 模式客户端使用 `Host.Send`，不使用 `Host.Request`。
4. 报告分别输出：
   - request/response req/s
   - notify msg/s
   - batch item/s
5. 在协议注册层增加校验：
   - `HasResponse=false` 的协议禁止通过 `Request()` 发送。
   - `HasResponse=true` 的协议禁止通过 `Send()` 发送，除非显式允许异步请求。

预期收益：

- 高频无状态消息不再进入 RequestManager。
- 减少 waiter channel、map lock、timeout、响应匹配和回包成本。
- 能快速判断当前瓶颈到底主要来自 RequestManager，还是来自 QUIC/加密/编码。

### 第二阶段：协议分类重构，建立三条数据路径

目标：从架构上减少错误链路使用。

任务：

1. 为 ProtocolSpec 增加 DeliveryMode 和 Reliability。
2. 将协议分为：
   - 控制面 request：握手、identify、find-node、关键指令。
   - 数据面 notify：心跳、peer hints、node status、交易广播、区块广播通知。
   - 同步面 batch/stream：headers、block、resource 拉取。
3. 将交易、区块、投票等可分区业务改成 state-key。
4. 为 query block/header/get resource 增加 batch 请求结构。

预期收益：

- 控制流保持正确性。
- 数据流吞吐明显提升。
- 同步流减少“一对象一请求”的放大成本。

### 第三阶段：热点拆锁

目标：降低 request/response 保留路径的锁竞争。

任务：

1. RequestManager shard 从 32 提升到 128 或 256，并通过压测选择最优值。
2. 将 host connection 表拆成 connection shard，减少全局 host.mutex 压力。
3. 对稳定连接增加 per-peer 轻量缓存，减少每次写前全局 RLock。
4. 对 DialPeer 增加或强化 singleflight，避免并发拨号风暴。
5. ordered sequencer 保持必要保序，但 state-key 协议必须按业务 key 分散。

预期收益：

- request/response 模式下 p95/p99 降低。
- 高并发时 peer backoff 和 timeout 概率下降。

### 第四阶段：内存与编码优化

目标：减少 GC 和短生命周期对象。

任务：

1. 对 message marshal buffer 使用 sync.Pool。
2. 对 AES-GCM seal/open 临时 buffer 使用 sync.Pool。
3. 对压测 payload buffer 使用 worker 本地复用。
4. 批量写入时尽量复用 frame header buffer。
5. 增加严格测试，防止 buffer 提前归还导致错包。

预期收益：

- 降低 heap alloc 和 GC CPU。
- 高频 notify 和 batch 场景收益更明显。

### 第五阶段：QUIC 写路径优化

目标：减少 stream 和 channel 热点。

任务：

1. QUIC stream pool 预热，避免高压中动态创建 stream。
2. 长生命周期保留常用 priority stream，减少 stream 注册/注销。
3. notify 高吞吐路径支持批量 drain，一次 worker 循环处理多条 queuedWrite。
4. 按优先级保留隔离，不能让低优先级大包阻塞高优先级共识消息。

预期收益：

- 提升无状态 notify msg/s。
- 降低高压下 streamMutex 和 channel 阻塞。

### 第六阶段：用画像闭环验收

每轮优化后必须记录：

- request/response req/s
- notify msg/s
- batch item/s
- p50/p95/p99
- failed/errors/payload mismatch
- CPU profile top10
- mutex profile top10
- block profile top10
- heap profile top10

验收标准：

- 正确性：failed=0，payload mismatch=0。
- 稳定性：10 分钟持续压测无泄漏、无明显抖动。
- 性能：notify msg/s 明显高于 request/response req/s。
- 锁竞争：mutex/block profile 中 RequestManager、host.mutex、streamMutex 占比下降。

## 12. 最终结论

当前 P2P 的问题不是大量错误数据竞争，而是 request/response 重链路在高频场景下产生了大量必要同步竞争。

根治方向不是单纯调大 worker 或 stream pool，而是：

1. 控制面继续 request/response。
2. 数据面改 notify。
3. 同步面改 batch/stream。
4. 有状态业务用 state-key 分区并行。
5. 剩余强确认路径再做 RequestManager、host connection 表、SecureSession、QUIC 写路径的热点拆锁。

这套方案可以同时保证正确性、吞吐和可维护性，是后续 P2P 模块继续提升 req/s 的主线。
