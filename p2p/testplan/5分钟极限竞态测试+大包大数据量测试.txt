4. 压测结束后是否有残留进程或资源未释放。

七、延迟与抖动分析

- p99 最高约 10.678s，最大延迟最高约 11.560s。
- 高尾延迟主要来自不限速、64 并发、全互联同时发送导致的 CPU、网络、QUIC 流控和用户态调度排队。
- 由于失败数为 0，不能把该抖动直接判定为代码错误。

16MB 大包测试：
- p99 最高约 2.94s，最大延迟最高约 3.03s。
- 在受控速率下延迟稳定，说明大包路径没有明显异常。
- 生产环境基本没有 16MB 数据包，该测试已覆盖极端边界场景。

八、连接修复效果分析

修复点验证目标：
连接替换时，旧连接关闭、重复连接淘汰或等待响应超时可能导致瞬态请求失败。修复后请求层对明确的连接层瞬态错误最多重试一次，并使用新的 request ID 和当前连接重新发送。

验证结果：
1. 1MB 极限竞态下总请求 25978 次，失败 0 次。
2. 16MB 大包下总请求 1197 次，失败 0 次。
3. warmup 连接竞争 WARN 未扩散成请求失败。
4. 高压力下没有再观察到连接替换导致的业务失败。

结论：
连接替换瞬态错误修复有效，当前结果支持进入更长时间稳定性测试阶段。

九、风险与建议

当前风险：
1. 5 分钟测试只能证明短时间高压稳定，不能完全替代 1 小时以上 soak test。
2. 1MB 不限速极限测试已出现明显尾延迟，生产必须配置发送速率、并发上限和背压策略。
3. 16MB 大包虽然稳定，但生产中不建议频繁发送该级别大对象，应优先分片、压缩或使用专用同步通道。
4. warmup 阶段仍有连接竞争 WARN，虽然不影响业务成功率，但日志上仍需要区分启动期噪声和运行期错误。

生产建议：
1. 普通业务消息建议控制在 1MB 以下。
2. 高优先级消息、普通消息、低优先级同步消息继续使用 QUIC 多 stream 隔离，避免大包阻塞关键消息。
3. 大包同步建议独立限速，并配置更小并发和更小写队列。
4. 对外网弱带宽节点应使用 1MB 或更小包测试，避免 16MB 大包把链路打满后产生误判。
5. 下一轮建议执行 30 分钟到 1 小时内网 soak test，观察 RSS、goroutine、p99、连接重试次数是否长期稳定。

十、最终判断

本轮 5 分钟内网极限竞态测试和 16MB 大包大数据量测试均通过。

代码层面未发现导致请求失败、内存泄漏或 goroutine 泄漏的问题。1MB 极限测试中的尾延迟抖动属于资源饱和表现；16MB 大包测试在受控参数下表现稳定，说明当前 QUIC 改造和连接修复具备继续扩大测试窗口的基础。
