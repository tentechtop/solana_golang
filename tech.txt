项目序列化规范

一、核心结论

本项目不使用 protobuf。

长期序列化分层如下：

1. P2P / 节点通信：P2P frame + Borsh message body
2. Solana 链上账户 / instruction data：Borsh
3. 外部 HTTP / 管理接口：JSON DTO
4. 业务内部事实模型：Go domain struct
5. 数据库存储：结构化字段优先，必要时保存 Borsh raw bytes
6. Hash / Sign：专用 canonical payload，不直接 hash 传输包 bytes


二、设计边界

Borsh 负责所有二进制协议字节。
JSON 负责外部接口展示和输入。
Go domain struct 负责业务一致性。

同一份核心业务事实只能以 domain model 作为权威来源：

json dto <-> domain model <-> borsh payload
p2p borsh message <-> domain model

禁止 JSON DTO 直接进入 P2P 或链上编码路径。
禁止任意 raw bytes 不带 schema_id 入库。


三、P2P 通信规范

P2P 使用两层结构：

1. P2P frame
   - magic
   - protocol_version
   - message_type
   - payload_length
   - checksum

2. P2P message body
   - Borsh 固定字段布局

当前 P2P message body 字段顺序：

version u16
id string
type u32
from_peer_id string
to_peer_id string
request_id string
flag u8
created_at_unix_milli i64
payload bytes

要求：

1. 单包必须有最大长度。
2. frame 必须先校验 magic、version、length、checksum。
3. Borsh 解码后必须 EnsureEOF。
4. 解码后必须执行业务 Validate。
5. message_type 必须走白名单注册。
6. 禁止 P2P 使用 JSON。
7. 禁止引入 protobuf。


四、Solana 链上数据规范

Solana 链上账户数据和 instruction data 统一使用 Borsh。

Borsh 使用要求：

1. 链上结构必须有 Version 字段。
2. 字段只能尾部追加，禁止中间插入。
3. 禁止修改已发布字段类型。
4. 删除字段必须保留占位或新增版本。
5. 禁止在核心链上结构中使用无序 map。
6. 数组、字符串、bytes 必须有最大长度。
7. 反序列化成功后必须继续做业务校验。
8. Go / Rust 必须建立 golden bytes 测试，保证字节级兼容。


五、外部 JSON 接口规范

外部 HTTP / 管理接口使用 JSON。

JSON 只作为输入输出 DTO，不作为系统事实数据格式，不参与核心 hash、签名、P2P、链上编码。

JSON DTO 必须显式转换为 domain model：

api dto <-> domain model


六、数据库存储规范

数据库以结构化字段为主，不得只依赖 opaque blob 完成核心查询。

推荐字段：

id
hash
type
version
codec
schema_id
slot
peer_id
status
created_at
updated_at
payload_bytes
payload_hash

raw payload envelope：

type
version
codec
schema_id
payload_length
payload_bytes
payload_hash

要求：

1. 核心检索字段必须拆列。
2. raw bytes 必须带 version、codec、schema_id、hash。
3. 写入数据库前必须完成输入校验和版本校验。
4. raw bytes 必须能通过 Schema Registry 找到对应解码器。
5. payload_bytes 写入后不得原地改写。
6. 协议升级必须通过新 version 或迁移任务完成。


七、Schema Registry 规范

项目必须维护 Schema Registry，用于管理 Borsh 历史版本解码规则。

Schema Registry 职责：

1. 通过 type、version、codec、schema_id 定位唯一 schema。
2. 注册当前版本和历史版本的 decode / validate / migrate 方法。
3. 禁止解析未注册 schema 的 raw bytes。
4. 禁止同一个 schema_id 指向不同结构。
5. 协议升级时必须新增 schema 记录，不能覆盖旧 schema。
6. 历史数据读取必须先按原版本解码，再显式迁移到当前 domain model。
7. 启动时必须校验关键 schema 已注册，缺失时直接失败。
8. Schema Registry 变更必须配套单元测试和 golden bytes。


八、Hash / Sign 规范

禁止直接使用传输包 bytes 作为安全 hash 或签名原文。

推荐规则：

hash = SHA256(domain_separator || message_type || version || canonical_payload)
sign = SIGN(domain_separator || message_type || version || canonical_payload)

说明：

1. 链上相关 hash 可以使用明确的 Borsh bytes。
2. P2P 去重、日志、缓存 key 可以使用消息 bytes hash。
3. 涉及资产、账户状态、共识、安全决策的 hash 必须使用专用 canonical payload。
4. hash/sign payload 必须包含 domain_separator、message_type、version。


九、代码组织规范

推荐目录职责：

api/dto       JSON DTO
p2p          P2P frame、连接、消息、协议分发
codec/borsh  Borsh 编解码基础能力
structure    业务事实模型和链上基础结构
database     结构化存储和 raw payload 存取
schema       Schema Registry 和版本迁移
utils        通用工具方法


十、开发铁律

1. 新增通信消息优先定义 domain model 和 Borsh 字节布局。
2. 新增链上结构必须定义 Borsh 编解码。
3. 新增外部接口必须定义 JSON DTO。
4. 新增核心业务状态必须先定义 domain model。
5. 不得引入 protobuf。
6. 不得将 JSON 作为 P2P 或链上核心协议。
7. 不得让链上 Borsh 结构承载外部 API 临时字段。
8. 所有反序列化入口必须做长度、版本、枚举、数组边界校验。
9. 所有底层 decode 错误向上传递时必须添加上下文。
10. 新增 raw bytes 存储必须同时注册 Schema Registry。
11. 协议升级必须保留历史解码能力。


十一、最终原则

通信简单性和确定性字节交给 Borsh。
外部可读性和接入体验交给 JSON。
业务一致性由 domain model 保证。
安全 hash 和签名由 canonical payload 保证。
历史 raw bytes 的可解析性由版本号和 Schema Registry 保证。


十二、ZK 零知识证明原生集成规划

核心定位：
将 ZK 作为 L1 原语原生集成，而非外挂 L2。
目标是把"高速透明链"升级为"高速+隐私+可扩容"的全能链。

一）为什么需要 ZK

1. 隐私缺失
   账本完全公开，地址余额、交易金额、历史记录全网可见：
   - 机构大额交易易被 MEV 抢跑、三明治攻击
   - 企业/金融场景（薪资、B2B 支付、稳定币发行）无法保密金额
   - 政务、医疗、身份类数据上链会泄露敏感信息
   ZK 实现"有效可证、内容不可见"：证明交易合法，不泄露金额、身份、余额。

2. 状态膨胀
   账户模型高速但状态膨胀快、存储成本高，长期扩容压力大：
   - ZK Compression：链下压缩状态，链上用 ZK 证明有效性，存储降 99%
   - ZK Rollup：大量交易链下处理，链上只存 proof，TPS 再提升一个量级

3. MEV 泛滥
   高速确认反而成为 MEV 攻击重灾区，大额订单在内存池被提前看见：
   - ZK 私密交易：订单/金额加密，内存池不可见，从源头防抢跑

4. 合规与身份
   传统公链无法兼顾"用户隐私"与"监管可审计"：
   - zkKYC：链下完成 KYC，链上用 ZK 证明"已合规"，不上传身份数据
   - 机密余额/转账：可开"审计密钥"，监管能查、公众不可见

5. 生态竞争
   ETH 生态 ZK Rollup 成熟，主打隐私与扩容；
   Solana 要保住高性能公链龙头，必须原生集成 ZK，而非外包给 L2。

二）核心落地场景

| 场景 | 技术方案 | 目标 |
|------|----------|------|
| 隐私交易 | ZK + ElGamal 加密，Token 原生支持机密转账与余额隐藏 | 隐藏金额与身份，仅证明交易合法 |
| ZK Compression | 链下压缩账户数据，链上提交 ZK 证明验证，更新 Merkle 根 | 存储成本降低 99%，租金降低 1000 倍 |
| ZK Rollup / zkSVM | 链下批量处理交易，链上仅验证 proof；SVM 链下执行，链上 ZK 证明结果 | TPS 再提升一个量级，突破单链算力上限 |
| 抗 MEV | 订单加密、链下撮合、链上清算，内存池无明文订单 | 从源头消除抢跑与三明治攻击 |
| zkKYC 与合规 | 链下 KYC 生成 ZK 证明，链上验证证明，不泄露身份证/护照数据 | 合规 DeFi、机构入场、去中心化身份 |
| 轻客户端 | 区块状态生成 ZK 证明，客户端验证 proof 即可确认 | 无需全量同步，轻量级验证链上数据 |

三）原生集成技术路线

1. ZK 密码学库
   - 引入 gnark（Go 原生 ZK 库），支持 Groth16 / Plonk 证明系统
   - 集成 alt_bn128 椭圆曲线运算、Poseidon 哈希函数
   - 证明生成在链下（客户端/RPC 节点），验证在链上（验证器）

2. ZK 系统调用（Syscall）
   在 SVM 运行时中注册以下系统调用，合约可直接调用：
   - alt_bn128_add        — 椭圆曲线加法
   - alt_bn128_mul        — 椭圆曲线标量乘法
   - alt_bn128_pairing    — 双线性配对
   - poseidon_hash        — Poseidon 哈希
   - groth16_verify       — Groth16 证明验证

   设计要点：
   - Groth16 验证仅需 17~50 万 CU，简单证明在默认指令预算内
   - 验证异步化，不阻塞主链出块
   - 保留亚秒级确认，速度不打折

3. Schema 扩展
   Borsh Schema 新增 ZK 相关类型：
   - ConfidentialTransfer  — 机密转账指令
   - ZkProof               — ZK 证明载体（proof bytes + public inputs）
   - MerkleProof            — 状态压缩的 Merkle 证明
   - ConfidentialBalance    — 机密余额（ElGamal 密文）
   - AuditKey               — 审计密钥（监管可查）

4. 状态压缩引擎
   数据库层支持 Merkle 树 + ZK 状态验证：
   - 链上：仅存储状态 Merkle 根哈希
   - 链下：RPC 节点存储完整账户数据
   - 交易时：客户端携带账户数据 + ZK 状态转换证明
   - 验证时：验证器验证 proof，更新 Merkle 根

5. 共识适配
   - 验证器在投票时额外验证 ZK 证明的有效性
   - ZK 证明验证作为交易执行的一部分，计入 CU 消耗
   - 无效证明的交易直接丢弃，不进入区块

6. Token 扩展
   Token 模块支持机密转账指令：
   - ConfidentialTransferInstruction::VerifyTransfer
   - 加密金额存于链上，仅持有解密密钥的方可查看
   - 支持"部分机密"（仅金额隐藏）和"全机密"（余额+金额+身份）

四）技术优势（与 ETH L2 方案对比）

| 维度 | Solana L1 原生 ZK | ETH ZK Rollup (L2) |
|------|-------------------|---------------------|
| 集成方式 | L1 系统调用，合约直接调用 | 独立 L2 链，需跨链桥 |
| 原子可组合性 | 是，ZK 验证与普通转账在同一交易 | 否，跨 L1/L2 存在延迟和风险 |
| 验证成本 | 极低（17~50 万 CU） | 较高（需 L1 提交 calldata） |
| 延迟 | 亚秒级，异步验证不阻塞出块 | 分钟级（L2 排序 + L1 最终确认） |
| 隐私粒度 | 灵活，可部分/全机密 | 通常全机密 |

五）实现阶段

阶段 1：密码学基础
  - 集成 gnark 库，实现 alt_bn128 曲线运算和 Poseidon 哈希
  - 实现 Groth16 证明的生成和验证

阶段 2：系统调用
  - 在 SVM 运行时中注册 ZK 系统调用
  - 实现 CU 计量和异步验证调度

阶段 3：机密转账
  - 实现 ElGamal 加密的机密余额和转账
  - 实现审计密钥机制

阶段 4：状态压缩
  - 实现链下状态存储 + 链上 Merkle 根
  - 实现 ZK 状态转换证明的生成和验证

阶段 5：高级特性
  - zkSVM 链下执行引擎
  - zkKYC 身份证明
  - ZK 轻客户端协议

六）安全要求

1. 证明系统必须经过第三方安全审计
2. 可信设置（Trusted Setup）的 toxic waste 必须安全销毁，保留可验证记录
3. 加密随机数必须使用密码学安全的随机源
4. 机密转账的审计密钥必须安全托管，支持多签和阈值恢复
5. 所有 ZK 相关代码必须经过形式化验证或充分的模糊测试
6. Schema Registry 必须注册所有 ZK 相关结构的版本历史
