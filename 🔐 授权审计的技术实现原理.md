# 🔐 授权审计的技术实现原理

授权审计的核心是 \\*\\*「密码学分级密钥体系 \+ 链上权限合约 \+ 链下解密验证」**的三层架构，本质是通过**密钥派生 \+ 门限密码学 \+ 零知识证明 \\*\\*，实现「平时完全隐私，审计时按权限定向解密」的平衡。

---

## 🧱 **核心密码学基础**

### 1\. **ElGamal 加密的天然特性：支持多密钥解密**

这是整个审计体系的基石：

```Plain Text
标准 ElGamal 加密：
密文 = (C1, C2) = (r·G, M + r·PubKey)

解密：
明文 M = C2 - PrivKey·C1

👉 关键特性：只要知道 PrivKey，就能解密。
👉 扩展：可以派生出多个不同权限的 PrivKey，分别解密不同范围的数据
```

**隐私余额加密结构（链上存储）**：

```go
type EncryptedBalance struct {
    L        [32]byte  // C1 = r·G （椭圆曲线点）
    R        [32]byte  // C2 = 余额 + r·UserPubKey （椭圆曲线点）
    AuditTag [32]byte  // 审计标记：Hash(审计公钥·r·G)，用于快速匹配审计权限
}
```

## 🔑 **三层密钥体系（从用户主密钥派生）**

所有密钥都从用户的**单个助记词**通过 BIP32 HD 钱包派生，用户只需要备份一套助记词：

```Plain Text
用户助记词 → BIP32 种子 → 主私钥 m
                          ↓
          ┌───────────────┼───────────────┐
          ↓               ↓               ↓
    m/44'/501'/0'/0'  m/44'/501'/1'/0'  m/44'/501'/2'/0'
          ↓               ↓               ↓
     SpendKey(签名)    ScanKey(扫描)     ViewKey(解密)
                                          ↓
                              ┌───────────┴───────────┐
                              ↓                       ↓
                    临时审计密钥派生          监管审计密钥派生
```

## 🎯 **分级授权审计的三层实现**

### **第一层：用户自主查看（ViewKey \- 基础权限）**

#### 实现原理：

1. **链上**：所有隐私余额用 `ViewPubKey` 加密

2. **链下**：用户用 `ViewPrivKey` 解密自己的余额和流水

3. **全程本地**：解密在用户钱包本地完成，ViewKey 永不上传链

#### 解密流程（用户钱包本地执行）：

```go
// 1. 从链上获取加密余额
encryptedBalance := GetEncryptedBalance(userAddress)

// 2. 用 ViewPrivKey 本地解密
C1 := encryptedBalance.L
C2 := encryptedBalance.R
S := ScalarMult(C1, ViewPrivKey)  // S = ViewPrivKey · C1
balance := SubtractPoints(C2, S)  // M = C2 - S

// 3. 解码明文余额
plaintextBalance := DecodeBalance(balance)
```

✅ **安全保证**：ViewKey 只在用户设备本地，全网节点和项目方都无法解密用户余额。

### **第二层：临时授权审计（第三方审计 \- 可控权限）**

#### 场景：

用户授权会计师、律师、合作方查看**特定时间范围、特定金额范围**的交易，**不是全部流水**。

#### 实现原理：**密钥派生 \+ 时间锁 \+ 范围证明**

```Plain Text
授权流程：
1. 用户生成「临时审计密钥」：
   AuditKey = HKDF(ViewPrivKey, 审计方公钥 + 起始时间 + 结束时间 + 金额上限)

2. 用户在链上授权合约中注册：
   - 审计方公钥
   - 授权时间范围 [startSlot, endSlot]
   - 授权金额上限
   - 临时审计公钥

3. 审计方解密：
   - 只能解密授权时间范围内的交易
   - 只能验证金额不超过授权上限
   - 授权到期自动失效
```

#### 技术细节：

1. **时间范围控制**：每笔隐私交易都包含 Slot 时间戳，解密时验证交易 Slot 在授权范围内

2. **金额范围控制**：用 ZK 范围证明，审计方只能验证「金额 ≤ 授权上限」，看不到具体数值

3. **权限可撤回**：用户随时可以在链上授权合约中删除审计方权限

4. **审计留痕**：所有解密操作都生成 ZK 证明上链，不可篡改

#### 示例：用户授权会计师查看 2026 年 Q1 的流水

```go
// 1. 用户生成本次授权的临时密钥
salt := Hash(审计方PubKey, 20260101, 20260331, 1000000)
tempAuditPrivKey := HKDF(ViewPrivKey, salt)
tempAuditPubKey := GetPublicKey(tempAuditPrivKey)

// 2. 链上授权合约注册
RegisterAuditPermission(
    auditor:    审计方地址,
    user:       用户地址,
    auditPubKey: tempAuditPubKey,
    startSlot:  2026年Q1起始Slot,
    endSlot:    2026年Q1结束Slot,
    maxAmount:  1,000,000 USDC,
)

// 3. 审计方解密（只能解密授权范围内的交易）
for _, tx := range GetUserTransactions(userAddr, startSlot, endSlot) {
    // 验证金额 ≤ 1,000,000（ZK证明，看不到具体值）
    VerifyRangeProof(tx.AmountProof, 0, 1000000)
    
    // 解密该笔交易金额
    amount := DecryptWithKey(tx.EncryptedAmount, tempAuditPrivKey)
}
```

✅ **安全保证**：

- 审计方拿不到用户的 ViewKey，只能解密授权范围内的交易

- 授权到期自动失效，用户随时可以撤回

- 审计过程全程留痕，可追溯

### **第三层：监管级审计（司法授权 \- 最高权限）**

#### 场景：

司法机关持合法文书，要求追溯特定地址的全部交易流水。

#### 实现原理：**门限密码学（TSS）\+ 多签授权**

```Plain Text
监管密钥不掌握在任何单一机构手中，而是拆分为 5 个分片：
├── 司法机关：1片
├── 金融监管：1片
├── 链治理委员会：2片
├── 独立第三方审计：1片

门限规则：(3,5) 门限 → 任意 3 个分片协同才能解密
```

#### 审计流程：

1. **司法申请**：法院出具调查令，指定需要追溯的地址和时间范围

2. **多签授权**：至少 3 个持分片的机构共同签名授权

3. **定向解密**：

    - 只能解密调查令指定的地址

    - 只能解密指定时间范围内的交易

    - 解密操作全程上链存证

4. **审计完成**：解密密钥立即销毁，不留后门

#### 技术实现：

```go
// 1. 监管公钥在链创世时写入，任何人都可以验证
const RegulatorPubKey = "..."  // 门限聚合公钥

// 2. 每笔隐私交易都用 RegulatorPubKey 额外加密一次审计标签
type PrivateTransaction struct {
    // 用户侧加密
    UserEncryptedAmount EncryptedBalance
    
    // 监管侧加密标签（只有监管门限密钥能解密）
    RegulatorAuditTag EncryptedAuditTag
}

// 3. 监管解密流程（需要3/5多签）
func RegulatorDecrypt(targetAddr Address, startSlot, endSlot Slot) {
    // a. 验证多签授权：至少3个监管机构签名
    VerifyThresholdSignature(authorization, 3, 5)
    
    // b. 验证调查令哈希上链存证
    VerifyCourtOrderHash(courtOrderHash)
    
    // c. 定向解密指定地址、指定范围的交易
    for _, tx := range GetTransactions(targetAddr, startSlot, endSlot) {
        plaintext := DecryptWithThresholdKey(tx.RegulatorAuditTag, thresholdSignatures)
        RecordAuditLog(tx.Hash, plaintext, authorization)
    }
}
```

✅ **安全保证**：

- 没有任何单一机构可以单方面解密用户交易

- 只能解密调查令指定的地址和范围，不能全网扫描

- 所有操作全程上链存证，防止权力滥用

- 没有永久后门，每次审计都需要重新授权

## ⚖️ **隐私与审计的安全边界**

### **链上 vs 链下的严格分工**

|操作|位置|可见性|
|---|---|---|
|加密余额存储|链上|全网可见密文，无人能解密|
|交易合法性验证|链上|ZK 证明验证，看不到明文|
|授权权限注册|链上|公开可见谁授权了谁，看不到解密内容|
|审计日志存证|链上|公开可见谁做了审计，看不到解密内容|
|明文解密|链下（本地）|只有密钥持有者能看到明文|
|ZK 证明生成|链下（本地）|用户 / 审计方本地生成，不上传原始数据|

### **防止权限滥用的机制**

1. **最小权限原则**：审计方只能拿到完成审计所需的最小数据范围

2. **权限可撤回**：所有临时授权都可以随时终止

3. **审计留痕**：所有解密操作都有不可篡改的链上记录

4. **时间限制**：所有授权都有到期时间，不能永久有效

5. **零知识验证**：审计方可以验证合规性，不需要看到全部明细

```go
type PrivateTransferInstruction struct {
    Sender      Address

    Amount      EncryptedBalance

// 链上验证逻辑（不需要解密）：
func VerifyPrivateTransfer(tx *PrivateTransferInstruction) bool {
    // 1. 验证 ZK 证明：交易合法
    VerifyGroth16Proof(tx.AmountProof)
    
    // 2. 验证签名：发送方拥有 SpendKey
    VerifySignature(tx.Signature, tx.SenderSpendPubKey)
    
    // 3. 不需要知道具体金额！
    return true
}
```

### 第四步：授权合约层

```go
// 实现链上授权合约：
contract AuditAuthorization {
    // 记录：用户 → 审计方 → 权限
    mapping(address => mapping(address => AuditPermission)) public permissions;
    
    function grantAuditPermission(
        address auditor,
        uint64 startSlot,
        uint64 endSlot,
        uint64 maxAmount,
        bytes32 auditPubKey
    ) external {
        permissions[msg.sender][auditor] = AuditPermission({
            startSlot: startSlot,
            endSlot: endSlot,
            maxAmount: maxAmount,
            auditPubKey: auditPubKey,
            active: true
        });
    }
    
    function revokeAuditPermission(address auditor) external {
        permissions[msg.sender][auditor].active = false;
    }
}
```
