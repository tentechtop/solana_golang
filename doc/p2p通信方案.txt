P2P 通信方案

版本：v1
范围：p2p、cmd 启动身份、节点发现、连接安全、消息收发、心跳和后续交易/区块广播接入。

一、核心决策

1. 不使用 libp2p。
2. 传输层使用 Go 标准库 TCP 和 quic-go。
3. 节点间消息统一使用 Borsh 或固定字段 Borsh 布局。
4. 连接建立后统一进入安全会话层，TCP 和 QUIC 都走同一套节点认证和 payload 加密。
5. QUIC 自带 TLS 只作为传输层加密，业务 payload 仍由安全会话层加密。
6. 中转节点未来只能读取外层路由字段，不能读取业务 payload。
7. 未来 0RTT 只允许幂等消息，交易、区块、投票必须等待握手确认。

二、配置模型

示例：

p2p:
  ip_type: "ip4"
  listen_ip: "0.0.0.0"
  listen_port: 5002
  default_protocol: "tcp"
  max_peers: 64
  network_id: "solana_golang:localnet:00000000000000000000000000000000"
  software_version: "solana_golang/0.1.0"
  min_outbound_peers: 8
  bootstrap_timeout_millis: 5000

说明：

1. 配置文件不再包含 peer_id。
2. cmd 启动时先从数据库读取节点身份。
3. 如果数据库没有节点身份，则通过 utils/ed25519.go 创建 32 字节公钥和 32 字节私钥 seed。
4. peer_id 使用公钥 base58 编码。
5. 公钥、私钥 seed 和 peer_id 必须一起持久化。
6. 私钥 seed 不得写入日志、RPC 返回、P2P 消息或错误文本。

三、模块分层

1. Transport 层

职责：
- 负责 TCP/QUIC listen、dial、read、write、close。
- 不做业务协议解析。
- 不关心交易、区块、共识和 DHT。

对应文件：
- p2p/transport.go
- p2p/tcp_transport.go
- p2p/quic_transport.go

2. Secure Session 层

职责：
- 连接建立后进行节点身份认证。
- 交换临时 X25519 公钥。
- 使用节点 Ed25519 私钥签名握手内容。
- 派生双向 AES-GCM 密钥。
- 加密和解密 Message.Payload。
- 维护单调 sequence，拒绝重放密文。
- 保存恢复票据，为未来 0RTT 做准备。

对应文件：
- p2p/secure_session.go

3. Message 层

职责：
- 定义统一 Message 结构。
- 定义 frame header。
- 执行 Borsh 编码、解码、checksum 校验和长度校验。
- 给协议注册表提供 message type。

对应文件：
- p2p/message.go
- p2p/protocol.go

4. Host 层

职责：
- 管理本地节点身份。
- 管理 transport 注册。
- 管理连接池。
- 管理 peer 表。
- 拨号时自动安全握手。
- 监听时自动安全握手。
- 发送消息时补齐 from_peer_id、to_peer_id、message_id、created_at。
- 处理 ping/pong 心跳。

对应文件：
- p2p/host.go

5. KAD 发现层

职责：
- 保存和查询邻近节点。
- 支持 find-node。
- 只负责发现和路由，不负责传输和业务消息。

对应文件：
- p2p/kad_bucket.go
- p2p/kad_distance.go
- p2p/kad_routing_table.go

四、连接建立流程

主动方流程：

1. Host 选择 peer 的地址。
2. 按协议优先级选择 QUIC 或 TCP。
3. Transport 建立原始连接。
4. Host 调用 SecureDialConnection。
5. 主动方生成 X25519 临时密钥对和 nonce。
6. 主动方构造 SecureSessionHandshake。
7. 主动方用节点 Ed25519 私钥签名握手内容。
8. 主动方发送 ProtocolSecureSessionV1 请求。
9. 主动方读取响应并验证签名、网络、协议版本和时间窗口。
10. 双方派生相同 session_id 和方向密钥。
11. Host 将 SecureConnection 写入连接池。
12. Host 保存恢复票据。

被动方流程：

1. Transport accept 原始连接。
2. Host 包装入站 handler。
3. Host 调用 SecureAcceptConnection。
4. 被动方读取 ProtocolSecureSessionV1 请求。
5. 被动方验证 peer_id、公钥、签名、network_id、协议版本和时间窗口。
6. 被动方生成本地临时密钥和 nonce。
7. 被动方返回 ProtocolSecureSessionV1 响应。
8. 双方派生相同 session_id 和方向密钥。
9. Host 将 SecureConnection 写入连接池。
10. Host 保存恢复票据。
11. 连接进入业务消息循环。

五、握手协商内容

安全握手必须包含：

1. version
2. role：initiator 或 responder
3. peer_id
4. identity_public_key
5. ephemeral_public_key
6. nonce
7. created_at_unix_milli
8. network_id
9. software_version
10. min_protocol_version
11. max_protocol_version
12. resumption_ticket_id
13. supports_zero_rtt_read
14. signature

强校验字段：

1. peer_id 必须等于 identity_public_key 的 base58。
2. signature 必须通过 identity_public_key 验证。
3. network_id 必须一致。
4. 协议版本范围必须有交集。
5. created_at_unix_milli 必须在允许时间窗口内。
6. ephemeral_public_key、nonce、signature 长度必须正确。

软协商字段：

1. software_version 需要记录，用于日志、监控、灰度策略和未来版本禁用策略。
2. 默认不因为软件版本不同而断开连接。
3. 如果未来出现严重漏洞版本，可以在 Host 层增加软件版本拒绝策略。

六、网络 ID 设计

network_id 是硬隔离字段，用于防止不同网络的节点误连。

推荐格式：

项目名:网络名:创世摘要

示例：

solana_golang:localnet:00000000000000000000000000000000

规则：

1. 本地开发网络、测试网络和生产网络必须使用不同 network_id。
2. network_id 不得为空。
3. network_id 不得包含控制字符。
4. network_id 不得只靠端口区分。
5. 如果未来创世块或初始配置改变，应改变 network_id。

七、消息格式

Message 字段：

1. id
2. type
3. from_peer_id
4. to_peer_id
5. request_id
6. flag
7. payload
8. created_at_unix_milli
9. version

Frame header 字段：

1. magic
2. version
3. message_type
4. payload_length
5. checksum

规则：

1. frame header 使用固定布局。
2. Message 使用 Borsh 编码。
3. checksum 对 Borsh 后的 Message bytes 计算。
4. payload_length 必须小于 max_message_size。
5. 解码后必须执行 Validate。
6. 未知 message type 不得进入业务 handler。

八、安全消息边界

安全连接只加密 Message.Payload。

不加密但参与认证的数据：

1. message id
2. message type
3. from_peer_id
4. to_peer_id
5. request_id
6. flag
7. created_at_unix_milli
8. version

这样做的原因：

1. 外层路由字段可以让 Host 和未来中转节点转发消息。
2. payload 仍然端到端加密，中转节点无法读取业务内容。
3. 外层字段作为 AEAD associated data 参与认证，攻击者不能把密文改挂到其他协议或其他请求关系上。

九、加密方案

身份密钥：

1. 节点长期身份使用 Ed25519。
2. 公钥 32 字节。
3. 私钥保存 32 字节 seed。
4. peer_id 等于公钥 base58。

会话密钥：

1. 每条连接生成新的 X25519 临时密钥。
2. 双方用 X25519 生成 shared secret。
3. 使用握手 transcript 派生 session_id。
4. 使用握手 transcript 派生两个方向密钥：
   - initiator -> responder
   - responder -> initiator
5. payload 使用 AES-256-GCM。
6. nonce 由方向 nonce 前缀和单调 sequence 组成。
7. 接收端要求 sequence 精确递增。

拒绝条件：

1. session_id 不匹配。
2. sequence 不等于期望值。
3. AES-GCM tag 验证失败。
4. payload 超长。
5. 会话过期。

十、0RTT 方案边界

当前阶段：

1. 握手完成后保存恢复票据。
2. 不开放任意 0RTT 业务发送。
3. 不持久化当前明文方向密钥。

未来阶段：

1. 恢复票据可以持久化到数据库。
2. 0RTT 只允许幂等消息。
3. 0RTT 消息必须带 ticket id、时间窗口和 replay nonce。
4. 接收端必须维护 replay cache。

允许 0RTT 的消息：

1. ping
2. 只读节点状态查询
3. 只读能力查询

禁止 0RTT 的消息：

1. 交易广播
2. 区块广播
3. 共识投票
4. 证书广播
5. 会改变状态或消耗资源的请求

十一、心跳机制

协议：

1. ProtocolPingV1
2. ProtocolPongV1

流程：

1. Host 定时向连接池中的 peer 发送 ping。
2. ping 使用 request message。
3. 对端收到 ping 后立即返回 pong。
4. pong 的 request_id 必须等于 ping 的 message id。
5. 收到任何有效消息都刷新 LastReadUnixMilli。
6. 成功写入消息刷新 LastWriteUnixMilli。
7. 连续失败次数超过阈值则关闭连接。
8. 空闲超过 ConnectionIdle 则关闭连接。

默认建议：

1. heartbeat_interval：15 秒。
2. connection_idle：45 秒。
3. max_peer_failures：3。

十二、连接池策略

连接池 key：

peer_id

写入时机：

1. 出站安全握手成功后写入。
2. 入站安全握手成功后写入。

替换规则：

1. 同一 peer_id 只能保留一个当前连接。
2. 新连接成功后关闭旧连接。
3. 连接关闭时移除 connection state。
4. Host 关闭时关闭所有连接并清理恢复票据。

ConnectionState 应记录：

1. peer_id
2. connection_id
3. protocol
4. local_address
5. remote_address
6. encrypted
7. network_id
8. remote_software_version
9. negotiated_protocol
10. connected_at_unix_milli
11. last_read_unix_milli
12. last_write_unix_milli
13. last_heartbeat_unix_milli
14. failure_count

十三、协议注册

所有业务消息必须先注册协议定义。

当前核心协议：

1. ping
2. pong
3. secure-session
4. find-node request
5. find-node response
6. peer-hints
7. node-status
8. receive-transaction
9. receive-block
10. query-block-by-hash
11. query-block-by-height
12. query-common-ancestor
13. hotstuff-vote
14. hotstuff-qc

规则：

1. 协议 ID 不得复用。
2. 协议 name 使用稳定路径格式。
3. 有响应协议必须通过 request_id 绑定原请求。
4. 无响应协议不得伪造 request_id。
5. handler 返回前必须完成 payload Validate。

十四、KAD 节点发现

目标：

1. 节点可以发现更多 peer。
2. 节点可以按距离查找邻近节点。
3. 节点可以保存 peer 地址和能力。

节点记录应包含：

1. peer_id
2. addresses
3. role
4. capabilities
5. protocol_version
6. software_version
7. latest_slot
8. block_height
9. best_block_hash
10. last_seen_unix_milli
11. last_connected_unix_milli
12. failure_count

find-node 流程：

1. 请求方发送 target_peer_id。
2. 接收方从 routing table 中查找最近 peer。
3. 接收方返回有限数量的 peer hints。
4. 请求方校验 peer_id 和地址归属。
5. 请求方写入本地 peer 表和 routing table。

边界：

1. DHT 不负责建立连接。
2. DHT 不负责加密。
3. DHT 不负责交易、区块和共识。
4. DHT 返回的地址必须经过 Host 拨号和安全握手验证。

十五、交易和区块广播接入

交易广播 payload：

1. version
2. transaction_bytes
3. created_at_unix_milli

接收流程：

1. SecureConnection 解密 payload。
2. Message 层完成 Borsh 解码。
3. 协议 handler 解析 transaction_bytes。
4. 执行 Transaction.Validate。
5. 执行签名验证。
6. 检查 recent blockhash。
7. 可选执行内存模拟。
8. 成功后进入交易池或临时队列。

区块广播 payload：

1. version
2. slot
3. block_hash
4. block_bytes

接收流程：

1. SecureConnection 解密 payload。
2. 解析 block_bytes。
3. 执行 Block.Validate。
4. 校验交易签名。
5. 校验 transactions root。
6. 可选重新模拟交易。
7. 成功后进入区块处理流程。

十六、错误处理

错误分类：

1. transport error：连接、读写、超时、关闭。
2. secure session error：身份、签名、网络、版本、加密、重放。
3. message error：frame、checksum、Borsh、长度、字段。
4. protocol error：未知协议、请求响应不匹配、handler 错误。
5. business error：交易、区块、共识校验失败。

要求：

1. 底层错误向上传递必须添加上下文。
2. 安全错误不得泄露私钥、shared secret、方向密钥和恢复 secret。
3. 入站握手失败应关闭连接。
4. 解密失败应关闭连接并记录 peer failure。
5. 业务 payload 校验失败只拒绝消息，不一定关闭连接。

二十一、结论

本系统的 P2P 通信应当采用“成熟传输 + 自研安全会话 + Borsh 消息 + Host 统一调度”的结构。

TCP 和 QUIC 不直接暴露给业务层。业务层只依赖 Host 和 Connection 接口。节点身份、网络版本、软件版本和密钥协商在连接建立阶段完成。交易、区块和共识消息只在安全会话建立后传输。未来引入中转节点时，外层路由可见，业务 payload 仍保持端到端不可读。
