# Solana Golang 架构与生产级代码规范

## 1. 目标

本文档用于约束项目后续架构设计、编码实现、测试验证和代码审查，目标是让代码长期保持生产级质量。

核心目标：

- 正确性优先：任何性能优化不得破坏协议语义、数据一致性和安全边界。
- 性能优先：关键路径必须减少不必要分配、锁竞争、阻塞 IO 和重复计算。
- 简洁可维护：只为真实复杂度建模，不做无需求支撑的过度抽象。
- 安全可控：所有外部输入必须校验，所有错误必须可追踪。
- 可测试可观测：核心逻辑必须有单元测试，关键路径必须可定位问题。

## 2. 分层架构

项目按职责分层，禁止跨层随意引用。

```text
cmd/          启动入口和进程装配
config/       配置文件和配置结构
api/          API 协议、DTO、对外接口定义
rpc/          RPC 服务、请求校验、响应转换
middleware/   RPC 和服务中间件
blockchain/   区块链核心业务逻辑
database/     本地持久化抽象和存储引擎实现
netsync/      网络同步流程
p2p/          P2P 网络协议和节点通信
poh/          PoH 相关逻辑
vm/           虚拟机和执行相关逻辑
monitor/      监控、指标、健康检查
structure/    领域结构体和基础模型
utils/        无业务状态的通用工具
doc/          架构、规范、设计决策和运维说明
```

依赖方向：

- `cmd` 可以装配所有模块，但不得承载业务逻辑。
- `rpc` 可以调用业务服务，但不得直接操作底层存储引擎。
- `blockchain` 可以依赖 `database.Database` 接口，不得依赖 Pebble、LevelDB 等具体实现。
- `database` 对上暴露稳定接口，对下通过引擎适配隔离具体存储。
- `utils` 不得依赖业务包，不得保存可变全局业务状态。

## 3. Database 架构规范

`database` 必须维持“业务接口 + 通用服务 + 引擎适配”的结构。

```text
database/
  interface.go          对外业务接口、配置、Table 定义
  factory.go            根据 Engine 创建实现
  errors.go             通用错误
  key_codec.go          表前缀、key 编码、边界计算
  cache.go              通用缓存
  kv.go                 底层 KVEngine 小接口
  service.go            基于 KVEngine 实现高级 Database 能力
  pebble/engine.go      Pebble KVEngine 实现
  leveldb/engine.go     LevelDB KVEngine 实现
```

强制规则：

- 上层业务只能依赖 `Database` 接口或 `NewDatabase` 工厂。
- 新增存储引擎必须实现 `KVEngine`，不得复制 `service.go` 中的分页、缓存、事务、范围查询逻辑。
- 不同引擎能力不一致时，必须显式返回 `ErrFeatureNotSupported`，禁止静默降级。
- 持久化 key 编码必须集中在 `key_codec.go`，禁止业务层自行拼表前缀。
- `Get` 返回的 `[]byte` 必须是拷贝，禁止暴露底层存储缓冲区。
- 批量写优先使用 `DataTransaction` 或 Batch，避免多次单写导致中间状态可见。

## 4. Utils 目录规范

`utils` 只能放无业务状态、无副作用或副作用明确的通用工具。

当前分组：

```text
encoding.go       Base58、Base64、Hex、short_vec 编解码
bytes.go          整数/字节转换、字节切片复制/拼接/反转
strings.go        通用字符串处理
crypto.go         随机数、安全比较、SHA、HMAC
bip39.go          BIP-39 助记词
ed25519.go        Ed25519 签名和验签
ecc_with_aes_gcm.go X25519、HKDF、AES-GCM
hd_wallet.go      HD 钱包派生
public_key.go     公钥、哈希、签名值对象
solana_keypair.go Solana 密钥对
multi_address.go  P2P multi-address
pda.go            Solana PDA
```

强制规则：

- 编码类工具统一放 `encoding.go`。
- 字节处理统一放 `bytes.go`。
- 字符串处理统一放 `strings.go`。
- 密码学基础能力统一放 `crypto.go`。
- 领域工具独立成文件，不得塞进通用工具文件。
- 工具函数必须无隐藏全局可变状态；确需全局状态时必须说明原因并保证并发安全。

## 5. 编码规范

必须遵循 Go 官方风格和 Google Go 风格精神。

强制规则：

- 所有 Go 文件必须通过 `gofmt`。
- 普通注释必须使用 UTF-8 中文，格式为“功能目的 + 实现原因”。
- `//go:embed` 等 Go 编译指令必须保持原样。
- 变量命名必须表意完整，禁止无意义缩写。
- 函数必须职责单一，核心逻辑应可单独测试。
- 代码嵌套层级不得超过 3 层；超过时必须提前返回或拆分函数。
- 禁止炫技式写法，优先直白可读。
- 优先使用标准库；新增第三方依赖必须说明收益、风险和替代方案。
- 不得提交无关格式化、无关重命名、无关大规模重构。

## 6. 错误处理规范

错误必须能定位问题发生位置和上下文。

强制规则：

- 底层错误向上传递时必须使用 `%w` 包装。
- 错误信息必须包含模块和动作，例如 `database: open engine: %w`。
- 可判定错误必须定义哨兵错误，例如 `ErrFeatureNotSupported`。
- 禁止吞掉错误，除非明确注释说明该错误可忽略。
- 禁止 `panic` 处理正常运行期错误；`panic` 仅允许用于包初始化不可恢复错误或测试辅助函数。
- 对外接口不得泄漏敏感信息，例如密码、私钥、完整 token。

## 7. 输入校验与安全规范

所有来自外部、网络、文件、配置、RPC 的输入都必须校验。

强制规则：

- 字节长度必须在使用前校验。
- 数组、切片、字符串索引前必须保证边界安全。
- 路径、地址、端口、协议枚举必须规范化后再使用。
- 密码学函数必须使用标准库或成熟算法实现，禁止自定义密码算法。
- 随机数必须使用 `crypto/rand`，禁止用 `math/rand` 生成密钥、nonce、seed。
- 私钥、助记词、seed 不得写入普通日志。
- 并发读写共享状态必须加锁或使用线程安全结构。

## 8. 性能规范

关键路径必须有明确性能意识。

强制规则：

- 热路径避免不必要的内存分配和字符串转换。
- 大表扫描必须使用迭代器或分页，禁止一次性全量加载。
- 批量写必须使用 batch 或 transaction。
- 锁范围必须尽量小，禁止持锁执行慢 IO、网络请求或用户回调。
- `[]byte` 返回值需要考虑所有权；可变数据必须拷贝。
- 缓存必须有容量上限或过期策略，禁止无限增长。
- 新增复杂算法前必须说明时间复杂度和空间复杂度。

## 9. 并发规范

所有并发操作必须可证明无竞态。

强制规则：

- 共享 map 必须受锁保护，禁止并发裸读写。
- goroutine 必须有退出条件，禁止泄漏。
- channel 必须明确关闭方，禁止多方关闭。
- context 必须向下传递到可能阻塞的 IO、网络和长期任务。
- 后台任务必须有日志、错误处理和停止机制。
- 单元测试中涉及并发的逻辑应使用 `go test -race` 验证。

## 10. 可观测性规范

生产问题必须能快速定位。

强制规则：

- 启动、关闭、连接外部资源、数据恢复、同步状态变化必须记录日志。
- 错误日志必须包含操作名、关键 ID、阶段和错误原因。
- 高频日志必须限流或降级为 debug，避免放大故障。
- 关键路径应暴露指标，例如请求数、错误数、延迟、队列长度、缓存命中率。
- 健康检查必须覆盖存储、网络、关键后台任务。
- 日志中禁止输出私钥、助记词、密码、token。

## 11. 测试规范

没有测试的核心逻辑不能视为完成。

强制规则：

- 新增导出函数必须有单元测试。
- 新增存储引擎必须通过数据库契约测试。
- 边界条件必须覆盖空输入、非法长度、极值、重复数据、不存在数据。
- 错误路径必须测试，不能只测成功路径。
- 涉及编码、签名、派生算法时必须使用标准测试向量。
- 修复 bug 必须先补能复现问题的测试。
- 提交前必须运行 `go test ./...`。

建议命令：

```powershell
go test ./...
go test -race ./...
```

## 12. 配置规范

配置必须显式、可校验、可追踪。

强制规则：

- 配置结构必须有清晰字段名和默认值策略。
- 必填配置必须在启动时校验。
- 配置错误必须阻止服务启动，禁止带错运行。
- 本地、测试、生产配置必须隔离。
- 敏感配置不得提交真实值。
- 配置变更必须说明兼容性和回滚方式。

## 13. API 与 RPC 规范

对外接口必须稳定、明确、可审计。

强制规则：

- RPC 入参必须做完整校验。
- 响应结构必须稳定，禁止随意改变字段含义。
- 错误码和错误消息必须可区分客户端错误、服务端错误和依赖错误。
- RPC 层只做协议转换和校验，不写核心业务逻辑。
- 不需要修改 `openapi.yml` 的需求，不得顺手修改该文件。

## 14. 代码审查清单

提交前必须自查：

- 是否只改了需求相关代码。
- 是否存在竞态、死锁、goroutine 泄漏。
- 是否校验了所有外部输入。
- 是否包装并保留了底层错误。
- 是否有核心路径日志或可观测信息。
- 是否避免了敏感数据日志。
- 是否覆盖主要场景和边界测试。
- 是否通过 `gofmt` 和 `go test ./...`。
- 是否没有引入不必要第三方依赖。
- 是否没有修改无关配置、文档或生成文件。

## 15. 新增模块流程

新增模块必须先明确职责边界。

流程：

1. 写清模块职责和不负责什么。
2. 定义对外接口和错误语义。
3. 实现最小可用版本。
4. 补充单元测试和边界测试。
5. 增加必要日志和健康检查。
6. 自查性能、安全、并发和可维护性。
7. 运行 `gofmt` 和 `go test ./...`。

## 16. 禁止事项

以下行为禁止进入主干：

- 未校验外部输入直接使用。
- 业务层直接依赖具体数据库引擎。
- 随意新增全局可变状态。
- 忽略错误或用 `_ = err` 掩盖错误。
- 在核心逻辑中使用魔法数字且无命名常量。
- 将测试辅助逻辑混入生产代码。
- 在日志中输出私钥、助记词、密码、token。
- 提交与需求无关的大面积格式化或重构。
- 为了“未来可能需要”提前设计复杂抽象。

## 17. 维护原则

项目长期维护以稳定、可理解、可验证为优先级。

当性能、简洁、安全、可维护性冲突时，排序如下：

1. 正确性和安全性
2. 数据一致性
3. 可观测性和可恢复性
4. 性能
5. 简洁和开发效率

任何偏离本文档的设计都必须在 `doc/` 中补充设计决策，说明背景、权衡、风险和回滚方案。
