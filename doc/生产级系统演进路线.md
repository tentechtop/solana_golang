# 生产级系统演进路线

## 一、当前判断

当前项目已经完成序列化治理方向的核心设计：

1. P2P / 节点通信使用 Borsh。
2. Solana 链上账户 / 指令数据使用 Borsh。
3. 外部 HTTP / 管理接口使用 JSON。
4. 业务事实以 domain model 为唯一权威来源。
5. 数据库 raw bytes 带 version、codec、schema_id。
6. Schema Registry 负责历史版本解析和迁移。
7. Hash / Sign 使用专用 canonical payload，不直接 hash 非 canonical bytes。

以上设计属于生产级协议地基，但系统整体还不能仅凭序列化规范称为生产级。

真正的生产级系统还必须补齐一致性、安全、P2P 容错、数据库可靠性、可观测性、测试体系和故障恢复。

## 二、一致性模型

目标：明确哪些数据必须强一致，哪些数据允许最终一致，避免读写路径语义不清导致状态分叉。

待完成工作：

1. 定义核心业务对象的一致性等级。
2. 明确写路径是否必须事务提交。
3. 明确读路径是否允许读取缓存或副本。
4. 定义 P2P 状态同步冲突处理规则。
5. 定义消息去重、重试、幂等机制。
6. 定义并发写入同一业务对象时的冲突检测策略。
7. 定义状态回滚、补偿、重放规则。

验收标准：

1. 每类核心数据都有明确的一致性说明。
2. 所有写接口具备幂等能力或明确拒绝重复提交。
3. P2P 重复消息不会造成重复写入。
4. 并发写入不会产生不可解释的最终状态。

## 三、安全模型

目标：保证节点身份、消息来源、重放防护和恶意输入处理具备生产安全边界。

待完成工作：

1. 定义节点身份模型。
2. 定义节点公钥注册、更新、吊销机制。
3. 定义 P2P 消息签名规则。
4. 定义 replay attack 防护规则。
5. 定义 nonce、timestamp、sequence 校验策略。
6. 定义 peer 黑名单和灰名单机制。
7. 定义 P2P 限流和恶意消息惩罚策略。
8. 定义外部 API 鉴权和权限模型。
9. 定义敏感操作审计日志。

验收标准：

1. 未认证节点不能参与核心 P2P 协议。
2. 篡改后的消息不能通过签名校验。
3. 重放旧消息不能改变系统状态。
4. 超大包、畸形包、高频请求会被拒绝或限流。

## 四、P2P 协议完整性

目标：P2P 不只是能收发消息，还要具备版本协商、流控、超时、背压和降级能力。

待完成工作：

1. 设计 handshake 流程。
2. 设计 protocol version negotiation。
3. 设计 message type registry。
4. 设计 max frame size。
5. 设计 checksum / payload_hash。
6. 设计 request timeout。
7. 设计 backpressure。
8. 设计 peer scoring。
9. 设计断线重连和退避策略。
10. 设计节点发现和节点淘汰策略。

验收标准：

1. 不兼容协议版本会被明确拒绝。
2. 单个慢节点不会拖垮本地节点。
3. 单个恶意节点不能通过刷消息耗尽内存。
4. P2P 消息类型必须经过白名单注册后才能处理。

## 五、数据库可靠性

目标：数据库既能支撑业务查询，也能支撑历史 raw bytes 重放、审计和迁移。

待完成工作：

1. 设计数据库 migration 机制。
2. 设计 schema version 管理流程。
3. 设计核心表索引。
4. 设计写入幂等键。
5. 设计 raw payload 校验任务。
6. 设计历史 raw bytes 重放工具。
7. 设计数据库一致性巡检任务。
8. 设计数据库备份和恢复流程。

验收标准：

1. 每次 schema 变更都有可执行 migration。
2. raw bytes 可通过 Schema Registry 稳定解码。
3. 历史数据可以迁移到当前 domain model。
4. 核心查询有明确索引，不依赖全表扫描。

## 六、可观测性

目标：系统出现异常时，可以快速定位是网络、协议、数据库、链上解析还是业务状态问题。

待完成工作：

1. 统一结构化日志字段。
2. 增加 P2P 消息收发 metrics。
3. 增加 P2P payload decode 失败率 metrics。
4. 增加 Borsh decode 失败率 metrics。
5. 增加 Schema Registry mismatch 告警。
6. 增加链上数据解析失败告警。
7. 增加数据库写入失败和慢查询监控。
8. 增加关键业务路径 tracing。
9. 增加节点 peer 状态面板。

验收标准：

1. 任意 decode 失败都能定位 message type、version、schema_id。
2. 任意 P2P 异常都能定位 peer、message type、错误原因。
3. 任意数据库慢查询都能定位 SQL、耗时、调用路径。
4. 链上解析失败会产生可告警事件。

## 七、测试体系

目标：用测试约束协议演进、并发安全、历史兼容和边界输入。

待完成工作：

1. 补齐核心 domain 单元测试。
2. 补齐 Borsh golden test。
3. 补齐 Borsh Go / Rust golden test。
4. 补齐 Schema Registry 历史版本测试。
5. 补齐 P2P frame 边界测试。
6. 补齐反序列化 fuzz test。
7. 补齐数据库 migration 测试。
8. 补齐历史 raw bytes 回放测试。
9. 补齐并发竞态测试。
10. 补齐安全攻击输入测试。

验收标准：

1. 协议结构变更会触发 golden test 差异。
2. 历史 raw bytes 在新版本代码中仍可解码。
3. 畸形输入不会 panic。
4. 并发测试不出现 data race。

## 八、故障恢复

目标：节点崩溃、重启、网络分区、链上数据缺失时，系统可以恢复到可解释状态。

待完成工作：

1. 设计节点重启恢复流程。
2. 设计未确认消息重放流程。
3. 设计本地状态重建流程。
4. 设计链上数据回补流程。
5. 设计 Schema Registry 升级失败回滚流程。
6. 设计数据库 migration 失败回滚流程。
7. 设计 P2P 断线后的状态补齐流程。
8. 设计异常状态人工干预工具。

验收标准：

1. 节点异常退出后可以重新启动并恢复状态。
2. 未确认消息不会永久丢失。
3. 本地状态可从数据库和链上数据重建。
4. 协议升级失败不会破坏历史数据。

## 九、执行优先级

第一阶段：协议安全和数据可恢复。

1. P2P frame。
2. message type registry。
3. Schema Registry。
4. raw bytes envelope。
5. decode / validate / migrate 流程。
6. Borsh golden test 和跨版本 raw bytes 回放测试。

第二阶段：一致性和数据库可靠性。

1. 写入幂等。
2. 状态冲突处理。
3. 数据库 migration。
4. 历史 raw bytes 回放。
5. 核心索引优化。

第三阶段：安全和可观测性。

1. 节点身份认证。
2. 消息签名。
3. replay 防护。
4. 结构化日志。
5. metrics。
6. tracing。
7. 告警。

第四阶段：容错和恢复。

1. 节点重启恢复。
2. 断线重连。
3. 状态补齐。
4. 数据回补。
5. 升级失败回滚。

## 十、最终目标

生产级系统必须满足以下条件：

1. 数据写入可追踪。
2. 历史数据可解析。
3. 协议升级可兼容。
4. P2P 异常可隔离。
5. 安全攻击可防护。
6. 系统异常可观测。
7. 节点崩溃可恢复。
8. 核心路径有测试兜底。

只有当以上能力形成闭环后，系统才可以称为生产级设计。
