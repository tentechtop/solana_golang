Solana Golang 开发手册

版本：v2
核心决策：本项目不再使用 protobuf。系统二进制序列化统一使用 Borsh 或基于 Borsh 的固定字段布局；外部接口使用 JSON DTO；内部事实模型使用 Go domain struct。


一、序列化总边界

1. Go domain struct 是系统内部唯一权威事实模型。
   P2P 消息、区块、交易、账户、指令、共识数据进入业务层后，都必须先落到 domain struct，再做校验和状态转换。

2. Borsh 是系统唯一二进制序列化格式。
   P2P 消息体、链上 account data、instruction data、区块头、需要持久化的 raw bytes，都使用 Borsh 或明确基于 Borsh 的固定字段顺序。

3. JSON 只用于外部 HTTP/RPC DTO。
   JSON 不参与 P2P、链上编码、共识 hash、签名原文和数据库 raw payload。

4. Schema Registry 管理所有 raw bytes。
   数据库保存二进制 payload 时，必须带 type、version、codec、schema_id、payload_hash。

5. Hash/Sign 使用 canonical payload。
   不得直接 hash 任意传输包。签名原文必须显式定义 domain_separator、message_type、version 和 canonical_payload。


二、为什么去掉 protobuf

本项目区块链协议当前不复杂，需要序列化的结构数量有限。保留 protobuf 会引入额外成本：

1. 需要维护 proto 文件、生成代码和生成工具链。
2. domain model、protobuf DTO、Borsh payload 三层转换会增加重复代码。
3. protobuf 适合复杂通信协议演进，但本系统当前更需要确定、简单、可审计的字节布局。
4. 对安全 hash、签名、链上字节来说，protobuf 不是天然 canonical 格式。

因此当前统一采用：

节点通信：P2P frame + Borsh message body
链上数据：Borsh
数据库 raw bytes：Schema envelope + Borsh payload
外部接口：JSON DTO
内部事实：Go domain struct


三、目录职责

cmd/
  启动入口、配置装配、核心 schema 注册、资源生命周期管理。

schema/
  Schema Registry、raw payload envelope、canonical hash。

codec/borsh/
  Borsh 基础编解码工具，包含固定整数、固定字节、动态 bytes/string、EOF 校验。

p2p/
  P2P frame、Borsh message body、TCP/QUIC transport、连接、心跳、协议分发。

structure/
  区块、交易、账户、指令、签名等 domain model。

api/、rpc/
  外部 JSON DTO、JSON-RPC 路由、请求校验、响应转换。

database/
  结构化字段存储、KV 抽象、事务、批量写、raw payload 存取。

utils/
  无业务状态的通用工具。

doc/
  架构、规范、开发流程、案例和审查规则。


四、标准开发流程

1. 先定义 domain model。
   新增业务必须先定义 Go struct、Validate、Clone 和必要状态转换。

2. 判断边界。
   - 节点间通信：P2P frame + Borsh message body。
   - 链上账户或 instruction data：Borsh。
   - 外部 HTTP/RPC：JSON DTO。
   - 数据库 raw bytes：Schema envelope + Borsh payload。
   - 安全 hash/sign：canonical payload。

3. 显式转换。
   - JSON DTO <-> domain model。
   - domain model <-> Borsh payload。
   - P2P Borsh message <-> domain model。

4. 注册 schema。
   只要新增 raw bytes 存储或历史解码能力，就必须注册 type、version、codec、schema_id、decode、validate、migrate。

5. 写边界校验。
   长度、版本、枚举、数组数量、索引、公钥、签名、peer_id、hash 都必须校验。

6. 写测试。
   必须覆盖正常路径、非法版本、非法枚举、超长 payload、hash mismatch、未注册 schema、历史版本迁移。


五、Borsh 使用边界

1. 必须使用 Borsh 的场景：
   - P2P message body。
   - P2P 业务 payload。
   - Solana instruction data。
   - Solana account data。
   - 区块头固定字段编码。
   - 需要跨语言字节一致的结构。
   - 数据库 raw payload。

2. 不使用 Borsh 的场景：
   - HTTP/RPC 请求响应展示，使用 JSON DTO。
   - 日志文本输出，使用结构化日志字段。
   - 高层业务状态流转，使用 Go domain struct。

3. Borsh 编码规则：
   - 整数使用小端。
   - bool 只能是 0 或 1。
   - 固定数组不写长度前缀。
   - bytes/string 使用 u32 长度前缀。
   - 动态容器必须有最大长度。
   - 解码后必须 EnsureEOF。
   - 解码后必须 Validate。

4. Borsh 结构演进规则：
   - 必须有 Version 字段。
   - 已发布字段顺序不得改变。
   - 已发布字段类型不得改变。
   - 新字段只能尾部追加。
   - 删除字段必须保留占位或新增版本。
   - 历史版本必须保留 decoder 和 migrate。


六、P2P 开发规范

1. P2P 使用两层结构：

   P2P frame:
   - magic
   - protocol_version
   - message_type
   - payload_length
   - checksum

   P2P message body:
   - Borsh 固定字段布局

2. 当前 P2P message body 字段顺序：
   - version u16
   - id string
   - type u32
   - from_peer_id string
   - to_peer_id string
   - request_id string
   - flag u8
   - created_at_unix_milli i64
   - payload bytes

3. P2P 接收流程：
   1. 读取固定 frame header。
   2. 校验 magic。
   3. 校验 protocol_version。
   4. 校验 payload_length。
   5. 读取 payload。
   6. 校验 checksum。
   7. 使用 Borsh 解码 message body。
   8. EnsureEOF。
   9. 执行 Message.Validate。
   10. 按 message_type 分发到协议处理器。

4. 新增 P2P message type：
   1. 在 p2p/protocol.go 定义 ProtocolID。
   2. 在 DefaultProtocolSpecs 注册协议名、响应语义和优先级。
   3. 定义 payload domain struct。
   4. 定义 payload 的 Borsh 编解码。
   5. 注册 Schema Registry。
   6. 补 frame、decode、handler、异常输入测试。

5. P2P 禁止事项：
   - 禁止使用 JSON 做 P2P 消息体。
   - 禁止无上限读取网络 payload。
   - 禁止未注册 message type 进入业务处理。
   - 禁止只解码不校验。
   - 禁止绕过 checksum。


七、数据库 raw bytes 规范

1. 数据库以结构化字段为主。
   核心检索字段必须拆列，例如 id、hash、type、version、codec、schema_id、slot、peer_id、status、created_at、updated_at。

2. raw bytes 只能作为审计、重放、排障和历史兼容用途。
   不得让业务查询完全依赖 opaque blob。

3. raw bytes 必须保存：
   - type
   - version
   - codec
   - schema_id
   - payload_length
   - payload_bytes
   - payload_hash

4. raw bytes 读取流程：
   1. 读取 envelope。
   2. 校验 payload_hash。
   3. 通过 type/version/codec/schema_id 查找 Schema Registry。
   4. 使用历史 decoder 解码。
   5. 执行历史 Validate。
   6. 显式 migrate 到当前 domain model。


八、Schema Registry 规范

1. Schema Registry 负责：
   - 唯一定位 schema。
   - 管理 decode、validate、migrate。
   - 拒绝未注册 schema。
   - 防止同一个 schema_id 指向不同结构。
   - 支持历史数据解码和迁移。

2. 新增 schema 必须定义：
   - Type：稳定业务类型，例如 p2p.message。
   - Version：从 1 开始递增。
   - Codec：borsh、json、canonical。
   - SchemaID：稳定唯一 ID，例如 p2p.message.borsh.v1。
   - Decode：字节到结构体。
   - Validate：业务校验。
   - Migrate：历史版本到当前 domain model。

3. 启动时必须注册核心 schema。
   缺少核心 schema 时节点必须启动失败。


九、Hash 和 Sign 规范

1. 安全相关 hash/sign 必须使用 canonical payload。

推荐格式：

hash = SHA256(domain_separator || message_type || version || canonical_payload)
sign = SIGN(domain_separator || message_type || version || canonical_payload)

2. 允许 hash Borsh bytes 的场景：
   - P2P 去重。
   - 日志 trace key。
   - 缓存 key。
   - 非安全审计摘要。

3. 必须使用 canonical payload 的场景：
   - 账户状态。
   - 资产变更。
   - 交易签名原文。
   - 共识投票。
   - 安全决策。


十、序列化案例：序列化一个区块

1. 区块权威模型：

   structure.Block

   它包含：
   - Header：共识和索引字段。
   - Transactions：交易体。
   - Meta：运行时元数据。

   Meta 不进入共识 hash，不进入区块确定性序列化。

2. 区块自身序列化：

   block.MarshalBinary()

   当前流程：
   - block.Validate()
   - BlockHeader 使用 Borsh 固定字段顺序编码
   - 写入 short_vec 交易数量
   - 按顺序追加每笔 transaction.MarshalBinary()
   - 不写入 BlockMeta

3. 区块通过 P2P 广播：

   structure.Block
       -> block.Validate()
       -> block.MarshalBinary()
       -> BlockBroadcast domain payload
       -> Borsh payload bytes
       -> p2p.Message Borsh body
       -> P2P frame

   BlockBroadcast 示例字段：
   - version u16
   - slot u64
   - block_height u64
   - block_hash [32]byte
   - parent_hash [32]byte
   - block_schema_id string
   - block_bytes bytes

4. 接收 P2P 区块：
   1. 校验 P2P frame。
   2. 解码 p2p.Message Borsh body。
   3. 校验 message_type 是 block。
   4. 解码 BlockBroadcast Borsh payload。
   5. 校验 schema_id。
   6. 解码 block_bytes。
   7. 执行 block.Validate。
   8. 重新计算 block.Hash。
   9. 重新计算 transactions root。
   10. 通过后写数据库。

5. 数据库存储区块 raw bytes：

   envelope, err := schema.NewEnvelope(
       "block.full",
       1,
       schema.CodecBorsh,
       "block.full.borsh.v1",
       blockBytes,
   )


十一、JSON DTO 规范

1. JSON 只用于外部 API。
2. JSON DTO 必须显式转换为 domain model。
3. JSON DTO 不得直接进入 P2P、Borsh、数据库核心写路径。
4. 外部输出不得泄露私钥、助记词、seed、token。


十二、并发和一致性规范

1. 共享 map 必须加锁。
2. goroutine 必须有退出路径。
3. context 必须传递到阻塞 IO。
4. 可变 []byte 对外返回必须拷贝。
5. 批量写必须使用 transaction 或 batch。
6. 读路径必须明确读取最新状态、快照还是缓存。


十三、测试要求

每次新增序列化能力至少覆盖：

1. domain Validate。
2. Borsh round trip。
3. Borsh golden bytes。
4. Schema Registry 注册。
5. 未注册 schema 拒绝。
6. raw envelope hash mismatch。
7. P2P frame magic/version/type/length/checksum。
8. 非法枚举。
9. 非法版本。
10. 超长 payload。
11. trailing bytes。
12. 历史版本 migrate。

常用命令：

go test ./...
go vet ./...
go test -race ./p2p ./schema ./database/...

Windows 本机搜索：

Get-ChildItem -Recurse -File | Select-String -Pattern "关键字"


十四、代码审查清单

1. 是否先定义 domain model。
2. 是否统一使用 Borsh 做二进制序列化。
3. 是否没有把 JSON DTO 当业务模型。
4. 是否新增或更新 Schema Registry。
5. 是否保存 type/version/codec/schema_id/hash。
6. 是否所有解码入口都有最大长度限制。
7. 是否解码后 Validate。
8. 是否 EnsureEOF。
9. 是否有 golden bytes 测试。
10. 是否保留历史版本 decoder。
11. 是否错误有上下文和 %w 包装。
12. 是否没有无关重构。


十五、禁止清单

1. 禁止引入 protobuf。
2. 禁止 P2P 使用 JSON。
3. 禁止链上数据使用 JSON。
4. 禁止 raw bytes 不带 schema_id 入库。
5. 禁止未注册 schema 仍然解码。
6. 禁止修改已发布 Borsh 字段顺序。
7. 禁止修改已发布 Borsh 字段类型。
8. 禁止无上限读取 payload。
9. 禁止解码成功后跳过 Validate。
10. 禁止安全签名原文使用传输包 bytes。
11. 禁止私钥、助记词、seed、token 写入日志。
12. 禁止并发裸读写 map。
13. 禁止 goroutine 无退出路径。


十六、完成定义

一个功能完成必须满足：

1. 业务模型清楚。
2. Borsh 字节布局清楚。
3. Schema Registry 已注册。
4. raw bytes 可追踪、可校验、可重放。
5. 所有输入有边界校验。
6. 错误有上下文。
7. 核心路径有测试。
8. go test ./... 通过。
9. go vet ./... 通过。
10. 没有违反禁止清单。

本项目未来的开发目标不是“代码能跑”，而是“协议简单、字节确定、历史可重放、错误可定位、安全边界可证明”。
