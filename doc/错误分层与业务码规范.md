# 错误分层与业务码规范

## 1. 目标

本文档约束项目中的内部错误、对外业务码和 RPC 错误映射，目标是让错误处理稳定、可观测、可扩展，并避免核心模块依赖 API/RPC 层。

核心原则：

- 内部错误分模块定义。
- 对外业务码统一放 `api/codes`。
- RPC 层负责错误映射。
- JSON-RPC 标准错误码和业务码必须分开。
- 对外业务码必须稳定，不能随内部实现随意变化。

## 2. 分层模型

错误处理分三层：

```text
内部模块错误
  blockchain.ErrBlockNotFound
  database.ErrKeyNotFound
  p2p.ErrPeerNotFound
        ↓
API 业务码
  api/codes.CodeBlockNotFound
  api/codes.CodeAccountNotFound
  api/codes.CodeInvalidTransaction
        ↓
RPC 协议错误
  JSON-RPC code: -32602
  JSON-RPC message: invalid params
  JSON-RPC data.business_code: INVALID_PUBLIC_KEY
```

强制规则：

- 核心模块不得 import `rpc`。
- 核心模块原则上不得 import `api/codes`。
- `api/codes` 只定义对外稳定业务码，不定义内部 Go error。
- `rpc` 可以 import `api/codes` 和业务模块，用于错误映射。
- 内部错误必须保留上下文，业务码只表达对外语义。

## 3. 内部错误规范

内部错误由各模块自己定义，放在模块自己的 `errors.go`。

推荐结构：

```text
blockchain/errors.go
database/errors.go
p2p/errors.go
netsync/errors.go
vm/errors.go
```

示例：

```go
package blockchain

import "errors"

var (
	ErrBlockNotFound      = errors.New("blockchain: block not found")
	ErrInvalidBlock       = errors.New("blockchain: invalid block")
	ErrInvalidTransaction = errors.New("blockchain: invalid transaction")
)
```

强制规则：

- 错误消息必须包含模块名前缀。
- 可判定错误必须使用哨兵错误。
- 传递底层错误必须使用 `%w` 包装。
- 不得将 HTTP 状态码、JSON-RPC 错误码、业务码写入核心模块错误。
- 不得在错误中包含私钥、助记词、密码、token。

正确示例：

```go
return fmt.Errorf("blockchain: load block %s: %w", blockHash, ErrBlockNotFound)
```

错误示例：

```go
return errors.New("BLOCK_NOT_FOUND")
return errors.New("-32602")
return fmt.Errorf("private key %s invalid", privateKey)
```

## 4. 对外业务码规范

对外业务码统一放在：

```text
api/codes/codes.go
```

推荐定义：

```go
package codes

type Code string

const (
	CodeOK                 Code = "OK"
	CodeInvalidParams      Code = "INVALID_PARAMS"
	CodeInternalError      Code = "INTERNAL_ERROR"
	CodeMethodUnavailable  Code = "METHOD_UNAVAILABLE"

	CodeBlockNotFound      Code = "BLOCK_NOT_FOUND"
	CodeInvalidBlock       Code = "INVALID_BLOCK"
	CodeAccountNotFound    Code = "ACCOUNT_NOT_FOUND"
	CodeInvalidPublicKey   Code = "INVALID_PUBLIC_KEY"
	CodeInvalidTransaction Code = "INVALID_TRANSACTION"
	CodePeerNotFound       Code = "PEER_NOT_FOUND"
)
```

业务码命名规则：

- 使用大写蛇形命名。
- 必须稳定，发布后不得随意重命名。
- 表达对外语义，不表达内部实现细节。
- 数量要克制，避免每个内部错误都暴露为独立业务码。
- 删除或重命名业务码必须提供兼容策略。

允许：

```text
BLOCK_NOT_FOUND
INVALID_TRANSACTION
ACCOUNT_NOT_FOUND
```

不允许：

```text
PEBBLE_ITERATOR_FAILED
GOROUTINE_TIMEOUT_17
DB_BATCH_COMMIT_INTERNAL
```

## 5. JSON-RPC 错误码规范

JSON-RPC 标准错误码放在：

```text
rpc/errors.go
```

标准错误码：

```go
const (
	CodeParseError     = -32700
	CodeInvalidRequest = -32600
	CodeMethodNotFound = -32601
	CodeInvalidParams  = -32602
	CodeInternalError  = -32603
)
```

强制规则：

- JSON-RPC `error.code` 必须使用数字。
- 业务码不得放进 JSON-RPC `error.code`。
- 业务码应放在 `error.data.business_code`。
- JSON-RPC `message` 使用稳定英文短语。
- 多语言文案不得影响 `code` 和 `business_code`。

推荐响应：

```json
{
  "jsonrpc": "2.0",
  "id": 1,
  "error": {
    "code": -32602,
    "message": "invalid params",
    "data": {
      "business_code": "INVALID_PUBLIC_KEY"
    }
  }
}
```

## 6. RPC 映射规范

RPC 层负责将内部错误转换为 JSON-RPC 错误和业务码。

推荐文件：

```text
rpc/error_mapper.go
```

推荐结构：

```go
package rpc

import (
	"errors"

	"solana_golang/api/codes"
	"solana_golang/blockchain"
)

func mapError(err error) *Error {
	if err == nil {
		return nil
	}
	if errors.Is(err, blockchain.ErrBlockNotFound) {
		return errorWithBusinessCode(ErrInvalidParams, codes.CodeBlockNotFound)
	}
	if errors.Is(err, blockchain.ErrInvalidTransaction) {
		return errorWithBusinessCode(ErrInvalidParams, codes.CodeInvalidTransaction)
	}
	return errorWithBusinessCode(ErrInternalError, codes.CodeInternalError)
}
```

辅助函数示例：

```go
func errorWithBusinessCode(base *Error, businessCode codes.Code) *Error {
	return &Error{
		Code:    base.Code,
		Message: base.Message,
		Data: map[string]any{
			"business_code": businessCode,
		},
	}
}
```

强制规则：

- handler 不得直接把内部错误原文完整返回给外部。
- 未识别错误统一映射为 `INTERNAL_ERROR`。
- 参数错误统一映射为 `INVALID_PARAMS`。
- 不存在类错误根据对象类型映射，例如 `BLOCK_NOT_FOUND`、`ACCOUNT_NOT_FOUND`。
- 映射逻辑必须有单元测试。

## 7. HTTP 状态码规范

JSON-RPC 协议层通常 HTTP 返回 `200`，错误通过 JSON-RPC error 表示。

例外：

- 非 POST 请求：`405 Method Not Allowed`
- 请求体超过限制：`413 Request Entity Too Large`
- 服务不可用或主动限流：可返回 `503` 或 `429`

强制规则：

- 业务失败不应使用 HTTP 500 表达。
- JSON 解析失败应返回 JSON-RPC `-32700`。
- 方法不存在应返回 JSON-RPC `-32601`。
- 参数错误应返回 JSON-RPC `-32602`。

## 8. 多语言规范

多语言只能发生在 API/RPC 边界层。

强制规则：

- 内部错误不做多语言。
- 业务码不做多语言。
- JSON-RPC 标准 `message` 保持稳定。
- 如需多语言，放在 `error.data.localized_message` 或上层 API response。
- 客户端优先根据 `business_code` 做翻译。

示例：

```json
{
  "code": -32602,
  "message": "invalid params",
  "data": {
    "business_code": "INVALID_PUBLIC_KEY",
    "localized_message": "无效的公钥"
  }
}
```

## 9. 测试规范

错误映射必须测试。

必须覆盖：

- 内部错误映射到正确业务码。
- 未知错误映射为 `INTERNAL_ERROR`。
- 参数错误映射为 `INVALID_PARAMS`。
- JSON-RPC 标准错误码不被业务码污染。
- 返回内容不包含敏感信息。

测试示例：

```go
func TestMapBlockNotFound(t *testing.T) {
	rpcError := mapError(blockchain.ErrBlockNotFound)
	if rpcError.Code != CodeInvalidParams {
		t.Fatalf("code = %d, want %d", rpcError.Code, CodeInvalidParams)
	}
}
```

## 10. 新增错误流程

新增错误必须按以下流程：

1. 在对应模块 `errors.go` 中定义内部错误。
2. 判断是否需要暴露为对外业务码。
3. 如需要，在 `api/codes` 中新增稳定业务码。
4. 在 `rpc/error_mapper.go` 中增加映射。
5. 增加单元测试。
6. 确认错误不会泄漏敏感信息。
7. 运行 `go test ./...`。

## 11. 审查清单

提交前必须确认：

- 内部错误是否放在所属模块。
- 业务码是否只放在 `api/codes`。
- RPC 标准错误码是否仍为数字。
- 是否存在核心模块 import `rpc` 或 `api/codes`。
- 是否使用 `%w` 保留底层错误。
- 是否对未知错误做了兜底映射。
- 是否添加错误映射测试。
- 是否避免返回敏感数据。

## 12. 结论

项目统一采用以下规则：

```text
内部错误分模块定义
对外业务码统一放 api/codes
RPC 层做错误映射
JSON-RPC 标准错误码和业务码分离
```

该规则是项目长期稳定 API 契约的基础，后续新增模块和接口必须遵守。
